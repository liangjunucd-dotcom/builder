# Aqara Builder · 商业模式与平台策略

> **Aqara Studio** = 空间本体数据基座。每一台 Studio 对应一个物理空间，以空间语义为核心，面向 AI Agent 提供统一语义层。  
> **Aqara Builder** = 面向全球 Builder 的开发者平台与社区。类 GitHub 形态——创作、发布、Fork/Remix、协作、交付。  
> **Aqara Life** = 面向终端用户的空间体验应用，承载 Builder 创作的应用插件与方案。  
> **核心主张**：服务 + 数据双轮驱动——以更好的物理空间体验、能耗优化、照护、安全、运维与自动化交付用户价值；将过程沉淀为结构化数据资产和可复用的平台能力。  
> **生态位**：Aqara 作为 Apple Home 最大生态接入者之一，空间本体增强 Apple Home 的语义与交付能力。

---

## 一、愿景：每个空间都值得被智能化

```
Builder 创作方案 → 发布到社区 → 全球 Builder Fork/Remix
→ 方案落入真实世界的每一个空间 → 空间产生数据
→ 数据反哺 AI → AI 帮助创作更好的方案 → 飞轮加速
```

**终极图景**：Aqara 不需要自己做所有项目——全球 Builder 在平台上创作并交付，Aqara 服务好社区、维护好基础设施即可。每一个方案最终都能落入现实世界的每一个空间里。

### 空间本体的战略意义

Studio 系统以**空间本体**为核心，统一语义层带来三个关键价值：

1. **跨项目可复用**：同一套空间语义、设备抽象、自动化逻辑，可在不同项目中复用
2. **跨行业可适配**：住宅 → 商业楼宇 → 医疗 → 酒店 → 城市空间，本体不变，行业插件适配
3. **跨平台可流转**：Builder 创作 → Site Manager 部署 → Design Center 设计 → Aqara Life 使用

## 二、商业模式全景：四层收入引擎

```
┌──────────────────────────────────────────────────────────────┐
│                     Aqara 商业模式全景                         │
├────────────┬─────────────────────────────────────────────────┤
│ L1 硬件层   │ Studio 硬件销售 + 配件 + 设备                    │
│            │ 入口：每卖出一台 Studio = 一个新空间 = 一个新用户   │
├────────────┼─────────────────────────────────────────────────┤
│ L2 平台订阅 │ Builder 账号订阅（Starter/Explorer/Builder/Master/Team） │
│            │ 核心：能力解锁 + 月送 Credits + 协同能力            │
├────────────┼─────────────────────────────────────────────────┤
│ L3 生态分润 │ Marketplace 抽成（模板/插件/方案交易）             │
│            │ Builder 创作 → 社区购买 → 平台抽成 15-30%          │
├────────────┼─────────────────────────────────────────────────┤
│ L4 行业方案 │ 行业解决方案授权 + 部署服务费 + 数据增值            │
│            │ 住宅/商业/医疗/酒店/城市空间的标准化方案包           │
└────────────┴─────────────────────────────────────────────────┘
```

### 各层的收入逻辑

| 层级 | 收入模式 | 毛利 | 规模化路径 |
|------|---------|------|-----------|
| **L1 硬件** | 一次性销售 | 中 | 渠道 + 安装商网络 |
| **L2 平台订阅** | MRR/ARR | 高 | 用户增长 × ARPU |
| **L3 生态分润** | 交易流水抽成 | 极高 | 社区规模 × 交易频次 |
| **L4 行业方案** | 项目制 + 授权费 | 高 | 行业复制 × 合作伙伴 |

**关键洞察**：L1 是入口（每台 Studio = 获客），L2 是基本盘（稳定现金流），L3 是飞轮（社区规模效应），L4 是天花板（行业纵深）。

## 三、核心模型：账号付费，空间消费，社区增值

| 概念 | 定义 | 关键点 |
|------|------|--------|
| **Account（账号）** | 统一身份，一人一号 | 付费主体；拥有方案、积分、Builder Profile |
| **Builder Profile（创作者档案）** | 账号级公开身份 | 作品集、声望值、排行榜、徽章 |
| **Plan（订阅方案）** | 账号级：Starter / Explorer / Builder / Master / Team (+ Enterprise) | 决定能力天花板 |
| **Workspace（工作空间）** | 统一工作上下文 | 能力上限 = Owner Plan；协作与项目的容器 |
| **Credits（积分）** | 账号级余额 + 空间池（Team+） | 资源消耗的统一货币 |
| **Solution（方案资产）** | 可发布、可交易、可部署的创作物 | 归创作者所有，可跨平台流转 |

**一句话**：方案和积分跟账号走，空间是协作上下文，方案是可流转的数字资产，社区是价值放大器。

## 五、订阅方案（Plan）

### 设计原则

每档 Plan 的核心付费驱动力不同——不是"一切都多一点"，而是"解锁不同阶段需要的核心能力"：

| Plan | 核心付费驱动 | 适合 |
|------|-------------|------|
| **Starter (Free)** | 体验 → 转化 | 体验者、学习者、围观者 |
| **Explorer** | 初步创作 + 私有模式 | 个人探索者、轻度用户 |
| **Builder** | 创作效率 + 社区身份 + 高级功能 | 独立 Builder、安装商 |
| **Master** | 全面掌控 + 客户管理 + 数据分析 | 专业创作者、高级服务商 |
| **Team** | 协同交付 + 大团队 + Credits Pool | 团队交付、服务商 |
| **Enterprise** | 组织管控 + 平台对接 | 大企业、区域服务商 |

**Starter 也有 500 积分/月**（~20 次 Build），确保新用户能体验 AI Build，降低转化门槛。  
**创建空间不扣积分**。Plan 限制空间数量已足够，积分专注于创作与消费场景。

### 降级策略

当用户从高级 Plan 降级到低级 Plan 时，遵循以下原则：

1. **当前周期保护**：降级后当前计费周期仍享受原方案的全部能力
2. **下一周期生效**：下一计费周期起按新方案计算
3. **超限处理**：
   - **Workspace 成员**：已添加的成员不会被自动移除，但 Owner 无法新增成员直到回到限额内。系统提示"成员数超出当前方案限额"
   - **Workspace 数量**：超出限额的 Workspace 进入「只读冻结」状态，可查看和导出，不能新增或编辑内容
   - **Projects**：超出限额的项目进入「只读」状态
   - **Studio 绑定**：超出限额的 Studio 进入「离线待迁移」状态，Owner 需手动选择保留哪些
   - **Assets 存储**：已上传的文件不会被删除，但无法新增直到存储使用量低于新限额
4. **积分处理**：已有积分余额不受降级影响，但月送积分从下一周期起按新方案额度发放
5. **功能限制**：AI 高级功能、Remote Studio、Service Hub 等功能在新方案不支持时立即停用

### 升级路径

```
Starter（探索学习 → 首次 Build）
  ↓ 想要 Private mode / AI Build
Explorer（初步创作 → 社区浏览）
  ↓ 想要高级功能 / 远程 Studio / 更多项目
Builder（独立创作 → 社区参与 → 小规模交付）
  ↓ 需要客户管理 / 数据分析 / Marketplace
Master（专业创作 → 高级服务 → 数据驱动）
  ↓ 需要团队协作 / Credits Pool / 大规模部署
Team（团队交付 → 服务商 → 社区商店）
  ↓ 需要 SSO / Site Manager / Design Center / 对公
Enterprise（组织管控 → 行业方案 → 平台对接）
```

## 六、Credits 积分经济体系

### 设计哲学

Credits 是平台内一切资源消耗的**统一货币**，也是 Builder 社区经济的**流通介质**。

```
积分流向：
Aqara → 月送积分 → Builder 账号
Builder → 消耗积分 → AI Build / 购买模板 / 购买插件
Builder → 发布方案 → 他人购买 → Builder 获得积分收入
Builder → 积分提现（达到门槛后）
```

### 6.1 账号维度：个人积分

- **归属**：账号级，一人一个余额
- **来源**：方案月送 + 充值 + 社区活跃奖励 + Marketplace 收入
- **用途**：AI Build、方案购买、插件购买、在空间内的消耗
- **展示**：侧边栏底部/个人中心，不随空间切换变化

### 6.2 空间维度：Workspace Credits Pool（Team/Enterprise）

**早期（Free/Pro）**：所有消耗走个人 Credits，简单直接。

**Team/Enterprise 引入 Workspace Credits Pool**：

```
Workspace Credits Pool 逻辑：
1. Owner 向 Pool 充值（从个人积分转入 / 直接购买）
2. 空间内消耗优先扣 Pool
3. Pool 不足时提示 Owner 充值，不自动扣成员个人积分
4. Owner/Admin 可设置每成员每日消耗上限
5. 每笔消耗记录关联：成员、项目、时间 → 可对账、可导出
```

**为什么需要 Pool**：
- 安装商服务多个客户，每个客户空间独立核算，才能给客户开账单
- 团队 Owner 需要控制预算，不是"我的钱随便花"
- 未来 Site Manager 场景：客户付费购买 Pool，安装商在 Pool 里干活

## 七、统一 Workspace 模型

### 设计原则

**不区分「个人空间」与「共享空间」**——所有空间统一为 Workspace。

1. Plan 已限制总空间数和每空间成员数，区分个人/共享是多余的
2. 注册自动创建 1 个默认 Workspace
3. 简化心智模型：空间就是空间，Plan 决定能力

### 空间能力

- 每个 Workspace 的能力上限 = Owner 的 Plan
- 任何 Workspace 都可以邀请成员（受 Plan 约束）
- 任何 Workspace 都可以创建项目、绑定 Studio
- 业务功能（客户管理、服务枢纽等）由 Owner 的 Plan 决定
- Team+ 的 Workspace 有独立 Credits Pool

### 空间与 Studio 的关系

```
Workspace（工作上下文）
  ├── Projects（方案项目）
  │     ├── Build 1（住宅自动化方案）
  │     ├── Build 2（商业能耗方案）
  │     └── ...
  ├── Studios（绑定的 Studio 设备）
  │     ├── Studio A → 物理空间 "张先生家"
  │     ├── Studio B → 物理空间 "9F 办公室"
  │     └── ...
  └── Members（协作成员）
```

每台 Studio 对应一个物理空间。Studio 绑定到 Workspace 后，Builder 可以基于该 Studio 的空间语义数据进行方案创作。方案脱离 Studio 也可以独立存在（作为模板），但部署到真实空间时需要绑定具体 Studio。

### 团队成员视角

**场景：Alice（Free）被邀请加入 Bob（Pro）的工作空间**

Alice 看到的：
- **个人中心**：她的积分（50）、方案 Free
- **空间切换器**：Bob's Team（角色：Member）
- **侧边栏**：根据空间能力（Pro 级别）展示功能
- **升级方案**：她自己的账号方案升级选项

Alice 看**不**到的：
- Bob 的积分余额、方案详情、计费明细

**逻辑自洽**：成员享受 Owner Plan 带来的空间能力，个人中心只展示自己的信息。

### Installer 多客户空间场景

典型状态：
- 自己有 Pro 账号 → 自己的工作空间
- 加入 5 个客户的工作空间（Member / Admin）

| 工作上下文 | Credits 消耗 | 账单归属 |
|-----------|-------------|---------|
| 自己的空间 | 自己的个人积分 | 自己 |
| 客户空间（Free/Pro） | 客户 Owner 的个人积分 | 客户 |
| 客户空间（Team+） | 空间 Credits Pool | 空间级对账 |

### 数据层保留

内部仍保留 `BuilderSpaceKind = "personal" | "team" | "company"` 标识创建来源，**UI 层不展示此区分**。

## 八、方案资产化与生命周期

### 核心思想

Builder 创作的每一个方案都是**数字资产**——它归创作者所有（不归 Workspace），可以发布、被购买、被 Fork、被部署到真实空间，也可以跨平台流转。

### 方案生命周期

```
Draft（草稿）
  ↓ Builder 完成创作
Ready（就绪）
  ↓ Builder 选择发布
Published（已发布到社区）
  ↓ 被他人 Fork / 购买
Forked / Purchased
  ↓ 被部署到真实 Studio
Applied（已应用到物理空间）
  ↓ 空间运行产生数据
Live（运行中）
```

### 方案属性

```typescript
interface Solution {
  id: string;
  ownerId: string;          // 归属账号（创作者），不归 Workspace
  workspaceId: string;      // 创作所在的 Workspace
  
  // 资产属性
  publishStatus: "draft" | "ready" | "published" | "archived";
  license: "free" | "paid" | "custom";
  price?: number;           // Credits 定价
  category: string;         // 行业分类
  tags: string[];
  
  // 社区属性
  forkCount: number;
  useCount: number;
  starCount: number;
  rating: number;
  forkedFrom?: string;      // Fork 来源
  
  // 部署属性
  deployments: Deployment[]; // 部署到了哪些 Studio/空间
  
  // 空间语义
  spaceType: "residential" | "commercial" | "healthcare" | "hotel" | "urban" | "other";
  requiredDeviceTypes: string[];
  requiredProtocols: string[];
}
```

### 方案的跨平台流转

| 平台 | 方案的角色 | 收费接入点 |
|------|-----------|-----------|
| **Builder** | 创作 + 发布 + 社区交易 | Credits（AI 消耗）+ Marketplace 抽成 |
| **Site Manager** | 部署 + 运维 + 远程管理 | 按站点数订阅（SaaS 模式） |
| **Design Center** | 方案设计 + 客户提案 + 3D 可视化 | 按项目/次收费 |
| **Aqara Life** | 终端用户使用 + 应用插件 | 应用内购 / 增值服务 |

**关键原则**：方案在 Builder 里创作，可以被授权到任意平台使用。Builder 是创作工具和社区，其他平台是使用和交付场景。

## 九、Builder 社区飞轮与激励体系

### 飞轮模型

```
                    ┌──────────────┐
```

## 十五、反滥用分析

### 场景 1：大量创建 Workspace 转售

**为什么不成立**：
- Plan 限制空间数 × 成员数 = 规模有限
- 消耗走 Owner 积分池 → 人越多耗越快 → 无套利空间
- 自然结果：升级（更多收入）或停止（自解）

### 场景 2：Free 账号批量注册薅积分

**为什么不成立**：
- 积分不支持跨账号转移
- Free 积分有限（1,000/月），不值得
- 未来加：手机验证、邮箱验证、设备指纹

### 场景 3：Marketplace 刷量

**防线**：
- 购买行为需消耗真实 Credits
- 异常交易检测（同 IP、同设备、短时间大量交易）
- 方案评分和使用量有真实部署校验

### 内建防线

1. **Plan 限制** → 不能无限扩张
2. **积分消耗** → 人越多耗越快 → 天然付费驱动
3. **Free 适度** → 够体验不够重度
4. **积分不可跨账号转移** → 杜绝搬运
5. **声望体系** → 作弊会被扣声望、影响排名
6. **未来可加**：实名认证、异常检测、活跃度审计

## 十九、插件生态蓝图与开发者体验

### 当前技术选择：方案三（静态 Native 构建）

基于现实评估，当前选择**方案三**：不同插件组合打包为多个可执行体。

```
开发者写插件代码 → 提交到 Builder 平台 → 审核 + 安全扫描
→ 进入插件池 → 用户在 Studio Cloud 选择插件组合
→ 云端按插件分组构建打包 → 用户下载升级 Studio 固件
```

### 三类插件、三种运行态

**关键洞察**：不是所有插件都需要编译进 Native 固件。按运行环境分三类：

| 类型 | 运行环境 | 加载方式 | 开发语言 | 示例 |
|------|---------|---------|---------|------|
| **System Plugin** | Studio Native 固件 | 编译打包（方案三） | Kotlin/Java → Native | 协议适配、底层算法、设备驱动 |
| **UI Plugin** | Studio Web Runtime / Aqara Life | WebView 沙箱加载 | TypeScript + React | 仪表盘、控制面板、App 页面、小组件 |
| **Cloud Plugin** | Builder 平台云端 | 容器/Serverless | Any（Python/TS/Go…） | AI 模型、MCP Server、分析算法、集成服务 |

**为什么这样分**：
- System Plugin 必须高性能 + 高安全 → 方案三（静态构建）是对的
- UI Plugin 不需要 Native 性能 → Web 技术更灵活、迭代更快、开发者门槛更低
- Cloud Plugin 运行在云端 → 不受 Studio 硬件限制，可以跑重计算

### 插件开发者的完整旅程

```
1. 注册 Builder 账号（Free 即可浏览 SDK 文档）
     ↓
2. 升级 Pro（解锁插件开发能力）
     ↓
3. 创建 Plugin Dev Workspace
     ↓
4. 选择插件类型（System / UI / Cloud）
     ↓
5. 使用 Studio SDK + Studio Simulator 开发与测试
     ↓
6. 提交到 Builder 平台 → CI/CD 自动构建 + 安全扫描
     ↓
7. 审核通过 → 上架 Marketplace（免费 / 付费）
     ↓
8. 用户安装 → 开发者获得积分收入 + 声望
     ↓
9. 持续迭代：Bug 修复、新版本、用户反馈
```

## 二十、本地 vs 云端：能力边界与商业逻辑

### 核心原则

```
本地 = 免费基础设施（让 Studio 值钱 → 卖更多硬件）
云端 = 付费增值服务（本地做不到或做不好的事 → 可持续收入）
```

**不是二选一，而是双层架构**：本地保证基础体验永远可用（断网也能用），云端提供超越本地能力的增强。用户自己选择需要哪些云端能力。

### 能力分层全景

| 能力维度 | 本地（Studio 免费） | 云端（Builder 付费） | 为什么这样分？ |
|---------|-------------------|-------------------|--------------|
| **设备控制** | ✓ 完整 | 远程中继（全 Plan 免费含基础） | 本地+远程控制都是基础体验；Plan 限 Studio 数量已自然限流 |
| **自动化** | ✓ 完整（规则引擎） | AI 自动化生成 | 本地规则引擎免费；AI 生成需要算力 |
| **场景管理** | ✓ 完整 | 云端同步、版本历史 | 本地场景免费；云端同步是增值 |
| **仪表盘** | ✓ 基础内置 | 社区仪表盘插件、AI 生成 | 内置免费；高级来自社区生态 |
| **数据** | ✓ 本地存储 7 天 | 长期存储、分析、报告 | 本地短期免费；长期分析需要云存储 |
| **API / MCP** | ✓ 本地 API + MCP Server | 云端中继、MCP Hub | 本地 API 免费建生态；云端中继收费 |
| **AI 助手** | ✓ 本地 AI 插件（用户自带 Key） | AI Build（Aqara 模型） | 本地 AI 做钩子；云端 AI 做价值 |
| **插件** | ✓ 已安装的插件运行 | 插件市场、开发环境 | 运行免费；交易和开发在平台 |
| **协作** | — | 多人协作、共享 | 协作天然是云端能力 |
| **备份** | — | 方案 + 配置云端备份 | 备份需要云存储 |

### 为什么本地要免费？

1. **卖硬件的飞轮**：Studio 本地能力越强 → 用户越愿意买 → 硬件收入（L1）
2. **开发者飞轮**：本地 API 免费 → 开发者愿意在上面建东西 → 生态变强
3. **信任飞轮**：本地运行不依赖云 → 用户信任 → 更愿意尝试付费云服务
4. **Matter 生态位**：免费本地能力 = 对 Matter 社区的贡献 → 生态领导力

### 为什么云端能收费？

1. **不可替代的价值**：AI Build 训练于社区数据、远程访问需要中继、协作需要实时同步——这些不可能纯本地实现
2. **边际成本合理**：云计算有真实的运行成本（GPU、存储、带宽）
3. **网络效应**：云端用户越多，AI 越好、社区越活跃、方案越丰富——形成正反馈

## 二十一、AI 分层策略：本地 AI × 云端 AI

### 为什么两边都要有？

```
本地 AI（Studio Plugin）         云端 AI（Builder Platform）
        │                                │
  Hook（钩子）                      Value（价值）
  "AI 真的有用！"                   "Aqara 的 AI 比我自己的好太多了"
        │                                │
        └───────── 转化路径 ──────────────┘
```

### 本地 AI：Studio AI Plugin

| 属性 | 说明 |
|------|------|
| **运行位置** | Studio 本地 |
| **LLM 来源** | 用户自己的 API Key（OpenAI / Anthropic / 本地 Ollama） |
| **成本** | Aqara 零成本（Token 费用用户自己承担） |
| **能力范围** | 自然语言控制设备、简单自动化生成、设备状态查询、故障诊断 |
| **数据范围** | 仅当前 Studio 的本地数据 |
| **安装方式** | 从插件市场免费下载 AI Assistant Plugin |
| **战略意义** | 让用户体验到 AI 的价值 → 转化到云端 AI Build |

**本地 AI 能做的**：
```
用户：「晚上 10 点后客厅自动关灯」
→ 本地 AI 生成自动化规则 → 直接在 Studio 中执行

用户：「卧室空调好像不太冷了」
→ 本地 AI 分析设备数据 → 给出诊断建议
```

**本地 AI 做不了的**（引导到云端）：
```
用户：「帮我设计一个三室两厅的完整智能方案」
→ 本地 AI：「这需要跨房间的综合设计，推荐使用 Builder 平台的 AI Build，
   它基于社区数千个真实方案训练，效果更好。→ [打开 Builder]」
```

### 云端 AI：Builder AI Build

| 属性 | 说明 |
|------|------|
| **运行位置** | Builder 平台云端 |
| **LLM 来源** | Aqara 训练的专用模型（基于社区数据 + 空间本体） |
| **成本** | 消耗 Credits（付费） |
| **能力范围** | 完整空间方案设计、3D 可视化、视频渲染、自动化编排、跨系统优化 |
| **数据范围** | 社区所有脱敏数据、行业最佳实践、设备兼容性知识图谱 |
| **战略意义** | 核心付费能力 + 数据飞轮 + 社区差异化 |

**云端 AI 的护城河**：
- **社区数据训练**：基于数千个真实项目训练，知道什么方案真的好用
- **空间本体理解**：深度理解设备关系、空间语义、行业规范
- **持续进化**：每个被部署的方案都在增强 AI 的能力
- **这是本地 AI 永远无法复制的**

### 商业逻辑

```
本地 AI Plugin（免费） → 体验 AI 价值 → 转化到 Builder 平台
→ 使用 AI Build（Credits） → 方案部署到 Studio → 空间数据反馈
→ AI 模型更好 → AI Build 更有价值 → 更多人愿意付费
```

**不需要担心本地 AI 侵蚀云端收入**——两者解决的问题完全不同。本地 AI 处理日常操作（"关灯"），云端 AI 处理创意设计（"给我设计一个节能方案"）。差距就像本地 Git 和 GitHub Copilot 的差距。

## 二十二、MCP 与开放 API 策略

### 设计原则

```
本地 API / MCP = 免费基础设施 → 构建开发者生态
云端 API / MCP = 付费增值服务 → 突破本地限制
```

### 本地开放能力（免费）

| 能力 | 接口 | 用途 |
|------|------|------|
| **设备控制 API** | REST + WebSocket | 第三方系统控制 Studio 设备 |
| **空间数据 API** | REST | 读取设备状态、传感器数据 |
| **自动化 API** | REST | 创建/修改/触发自动化规则 |
| **MCP Server** | MCP 协议 (WebSocket) | AI Agent 与 Studio 交互 |
| **事件订阅** | WebSocket | 实时设备事件推送 |
| **插件 API** | 内部 SDK | 插件间通信与扩展 |

**为什么必须免费**：
- MCP/API 是 **基础设施层**，不是产品层
- 免费 = 开发者愿意投入时间 → 生态壮大 → Studio 更有价值
- 收费 = 开发者转向竞品 → 生态萎缩 → 战略失败
- 类比：Linux kernel 免费 → 生态繁荣 → Red Hat 靠服务赚钱

### 云端增值能力（付费 Credits）

| 能力 | 说明 | 计费方式 |
|------|------|---------|
| **Cloud Relay（远程中继）** | 不在局域网时，通过云端远程访问 Studio。基础远程控制全 Plan 免费（含 Free）；Plan 通过 Studio 绑定数限流（Free=2, Pro=30, Team=100）。Pro+ 享受优先通道和低延迟 | 基础免费（所有 Plan）；优先通道（Pro+） |
| **MCP Hub（MCP 市场）** | 社区开发的 MCP Server 集合：天气、日历、音乐… | 按安装计费 / 订阅 |
| **Aqara LLM（空间语义大模型）** | 针对空间本体微调的 LLM，理解设备和自动化 | Credits / 按 Token |
| **Cloud Compute** | 为 Cloud Plugin 提供运行环境（Serverless） | Credits / 按调用 |
| **Cloud Storage** | 方案备份、数据长期存储、版本历史 | 订阅（Pro+ Plan 包含基础额度） |
| **Cloud Analytics** | 能耗分析、使用趋势、AI 优化建议 | 订阅（Pro+ Plan） |
| **Enterprise API** | 高 QPS、SLA 保证、专用通道 | Enterprise Plan |
| **私有云部署** | 企业内网部署 Aqara 云服务 | Enterprise 合同 |

### MCP 生态的特别机会

MCP（Model Context Protocol）是连接 AI Agent 与工具的标准协议。Aqara 的战略机会：

```
Studio 作为 MCP Server → 全球 AI Agent 都能控制智能空间
→ Aqara Studio 成为 "AI 时代的空间接口标准"
```

| 角色 | Aqara 的位置 | 商业价值 |
|------|-------------|---------|
| **MCP Server 提供者** | Studio 本地免费提供 MCP Server | 生态入口 |
| **MCP Hub 运营者** | Builder 平台汇聚社区 MCP Server | 交易抽成 |
| **MCP 客户端消费者** | AI Build 通过 MCP 调用外部能力 | 增强 AI Build 价值 |

### 用户的选择权

**让用户自己选**，不替他们做决定：

```
DIY 用户（爱好者）：
  本地 API + 本地 AI（自带 Key）→ 完全免费 → 但要自己折腾

效率用户（Installer）：
  本地 API + 云端 AI Build + 云端 Relay → Pro 订阅 → 省时间赚更多

企业用户（服务商）：
  全套云端能力 + Enterprise API → Team/Enterprise → 规模化交付
```

### 平台双轨模型：Space Package × Capability Asset（v3.2 重新定义）

#### 核心定义

**Aqara Builder 是一个以 AI Build 为入口、以 Playable Space 为内容单元、以 Marketplace 为能力分发中心、以 Studio 为运行载体的空间创作与开发平台。**

用户不是在配置设备、画页面、写自动化——**用户是在把一个想法，变成一个可被看见、可被体验、可被部署、可被传播的空间方案。**

> **AI Build: Turn ideas into playable spaces.**

#### 两类核心单元

平台中存在两类本质不同的「东西」，这个区分必须非常明确：

**第一类：Space Package（内容与方案）**

| 属性 | 说明 |
|------|------|
| 定义 | 面向「体验与方案」的内容单元 |
| 包含 | Space Schema、布局、设备映射、自动化设计、页面/大屏、AI 行为、安装引导、预览内容 |
| 解决 | **"这个空间要做成什么样"** |
| 流通 | 在 Explore / Community 中传播和复用 |
| 价格 | 免费 Fork / 付费模板 |

**第二类：Capability Asset（能力与扩展）**

| 属性 | 说明 |
|------|------|
| 定义 | 面向「能力供给」的功能单元 |
| 包含 | JS 协议插件、Skill、数据处理模块、自动化脚本卡片、应用能力、连接器/驱动 |
| 解决 | **"这个 Studio 能做什么"** |
| 流通 | 在 Marketplace Store 中交易和分发 |
| 价格 | 免费 / 付费订阅 / 一次性购买 |

#### Capability 细分类型

| 类型 | 说明 | 举例 |
|------|------|------|
| **Protocol Capabilities** | 协议接入类 | JS protocol plugin、connector、driver |
| **Logic Capabilities** | 逻辑增强类 | 自动化脚本卡片、数据处理模块、规则引擎组件 |
| **AI Capabilities** | AI 相关能力 | Skill、语义处理、空间理解、诊断能力 |
| **Experience Capabilities** | 体验增强类 | App module、UI widget、interactive component、App Theme |

#### 能力流转链路

**能力不是项目文件的一部分，而是平台级资产。**

```
Marketplace → Assets → Assign to Studio → Enabled for Spaces
```

1. 能力首先存在于 **Marketplace**
2. 用户购买/获取后进入 **Assets**（用户的资产库）
3. 用户将资产 **assign 给某个 Studio**
4. Studio 为其下的一个或多个 **Spaces** 提供运行能力

为什么要这样设计：
- **资产归属清晰**：同一账号下统一管理，可查看授权/版本/到期状态
- **Studio 是运行载体**：一个 Studio 可能承载多个 Space，能力装在 Studio 上更合理
- **空间与能力解耦**：Space Package 可以声明依赖的 Capabilities，但不自带能力
- **商业化自然**：售卖的是「可购买、可授权、可安装、可升级的空间能力资产」

#### 三角关系

> **Space defines the experience. Capability enables the experience. Studio runs the experience.**

- **Space** 决定你想做什么空间体验
- **Capability** 决定这个空间有没有足够的能力实现它
- **Studio** 决定这些能力在哪个运行载体上真正执行

#### App Blueprint 与 Aqara Life 体验层（v3.3 新增）

Space Package 中的 App 设计以 **App Blueprint** 形式存在——声明式组件树 + 数据绑定 + 交互逻辑，而非重型应用代码。

```
Builder 创作 Space Package（含 App Blueprint）
  → 部署到 Studio（运行 + 数据接口）
  → Aqara Life 连接 Studio → 获取 Blueprint + 实时数据 → 渲染体验
  → 用户可切换 Theme（视觉皮肤）或用 AI 生成个性化 Theme
```

**三层职责划分**：
- **Builder**：定义空间的「骨骼」— 方案 Schema + 自动化 + App Blueprint
- **Aqara Life**：提供空间的「皮肤」— Theme 渲染 + 个性化 + 微调能力
- **Studio**：充当空间的「心脏」— 运行 + 数据 + 能力执行

**新创作者类型**：Theme Designer（门槛最低，只需审美能力，可在 Theme Store 销售皮肤）

**收入延展**：Builder Marketplace（方案 + 能力）+ Life Theme Store（App 皮肤 + Widget）+ AI Theme 生成（Credits 消耗）

> 详细分析见 [Aqara-Life-App架构与互动UGC策略分析.md](./Aqara-Life-App架构与互动UGC策略分析.md)
