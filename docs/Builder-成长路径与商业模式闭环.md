# Builder 成长路径与商业模式闭环

> 基于《Aqara Space Intelligence Platform 白皮书》战略框架，定义 Builder 平台的能力边界、成长路径、与 Studio 的分工协作关系，以及商业模式闭环。

---

## 一、Builder 的定位：云端控制面 + 生态面

Builder **不是** Studio 的在线版，也不是单纯的开发者社区网站。

| 维度 | Studio (本地) | Builder (云端) |
|------|--------------|---------------|
| **核心角色** | 空间操作内核（本地执行心脏） | 云端控制面 + 生态面 |
| **数据位置** | 本地图数据库、实时设备状态、事件流 | 方案资产、项目协作、模板市场、分析聚合 |
| **用户** | 现场安装师、系统集成商 | Builder 开发者、方案设计师、团队 Leader、远程运维 |
| **核心能力** | 设备接入、状态同步、Actions 执行、Functions 运行 | 方案创作、项目管理、选品采购、协作分发、运营分析 |

**一句话**：Studio 负责「空间里发生什么」，Builder 负责「空间怎么被设计、交付、管理和复制」。

## 二、核心流程：创作 → 方案 → 项目 → Studio

### 2.1 流程验证

当前的「创作 → 方案 → 项目」流程与白皮书战略**高度一致**：

```
创作（AI Build / 手动设计）
  ↓ 生成
方案（可复用的空间智能蓝图）
  ↓ 实例化
项目（绑定具体客户、站点、Studio）
  ↓ 下发
Studio（本地执行、设备入网、自动化运行）
  ↓ 回流
Builder（运维数据、服务结果、方案优化迭代）
```

### 2.2 各环节的价值锚点

| 环节 | 价值 | 商业锚点 |
|------|------|---------|
| **创作** | AI 降低设计门槛，自然语言生成空间+自动化 | AI Build 消耗积分 |
| **方案** | 可复用蓝图 = 数字资产，可分享/交易 | 模板市场（发布/购买） |
| **项目** | 完整交付生命周期管理 | Pro/Team 方案的核心能力 |
| **Studio 下发** | 从设计到现实的最后一公里 | 硬件入口 + 交付服务费 |
| **数据回流** | 运维优化、服务结果、行业洞察 | 增值分析服务 |

### 2.3 缺失环节补充

当前流程需要补充的：

1. **方案版本管理**：方案应有版本历史，支持 fork、diff、merge
2. **方案与项目的双向同步**：项目中的优化可以反哺方案模板
3. **服务结果建模**：项目交付后，需要记录「结果是否达成」而不只是「动作是否成功」
4. **多站点复制**：一个方案可以批量部署到多个 Studio/站点

## 三、Builder 上应做哪些能力

### 3.1 空间设计（2D/3D）

**在 Builder 上做**：
- 户型绘制 / 导入（2D 平面图）
- 3D 等轴视角预览（房间高度、家具、设备布局可视化）
- AI 户型生成（从图片/文字描述生成户型）
- 设备放置与覆盖范围可视化（信号、感应、触发区域）
- 方案模板的空间结构定义

**留给 Studio 做**：
- 真实设备的实时状态渲染（Live Twin）
- 精确的物理空间建模（点云、BIM 集成）
- 设备入网后的真实拓扑

**分工逻辑**：Builder 做**设计态**的空间可视化，Studio 做**运行态**的空间数字孪生。Builder 的 3D 是「这个空间应该长什么样」，Studio 的 3D 是「这个空间现在什么状态」。

### 3.2 自动化设计

**在 Builder 上做**：
- AI 对话式自动化意图描述 → 生成自动化流程
- 可视化流程编辑器（触发 → 条件 → 动作）
- 自动化模板（可复用的逻辑片段）
- 自动化预览 / 模拟（不需要真实设备也能看效果）

**留给 Studio 做**：
- 真实设备事件的自动化执行
- 本地低延迟规则引擎
- 设备状态变更触发的实时响应

**分工逻辑**：Builder 做「自动化的设计与蓝图」，Studio 做「自动化的执行与运行」。

### 3.3 AI Build

AI Build 是 Builder 的**核心差异化能力**，整个 AI 链路应在 Builder 上完成：

| AI 能力 | 说明 | 积分消耗 |
|---------|------|---------|
| **AI 户型生成** | 从文字/图片生成 2D 户型 + 3D 预览 | 200–500 |
| **AI 设备推荐** | 根据户型 + 意图自动推荐设备选品 | 100–300 |
| **AI 自动化生成** | 从自然语言生成自动化流程 | 100–500 |
| **AI 方案优化** | 基于运维数据优化已有方案 | 200–1,000 |
| **AI 3D 渲染** | 生成高质量 3D 空间效果图 | 500–2,000 |

### 3.4 选品与采购

**在 Builder 上做**：
- 设备目录浏览（分类、搜索、对比）
- 项目选品清单（BOM）管理
- 一键采购 / 跳转 Aqara Store
- 基于方案自动生成 BOM
- 设备兼容性检查

### 3.5 项目协作与交付

**在 Builder 上做**：
- 多人协作（Owner/Admin/Member）
- 项目生命周期管理（规划→采购→调试→配置→运维）
- 交付任务看板
- 客户管理（CRM 基础）

## 四、商业模式闭环

### 4.1 Builder 的收入来源

```
        ┌─────────────────────────────────┐
        │         Builder 收入矩阵         │
        ├─────────┬───────────┬───────────┤
        │  订阅费  │   积分消耗  │  交易抽成  │
        │(Plan)   │(AI Build) │(市场)     │
        ├─────────┼───────────┼───────────┤
        │Free/Pro │AI 户型生成 │方案模板交易 │
        │/Team    │AI 自动化   │插件市场    │
        │         │AI 3D渲染   │设备采购佣金 │
        │         │AI 方案优化 │增值服务    │
        └─────────┴───────────┴───────────┘
```

### 4.2 价值闭环

```
Builder 创作能力
    ↓ 降低设计门槛
更多方案被创建
    ↓ 方案可复用
更多项目被交付
    ↓ 每个项目需要设备
更多硬件销售（Aqara Store）
    ↓ 每个 Studio 产生运维数据
更多空间数据回流
    ↓ 数据训练 AI + 优化模板
更好的 AI Build 能力
    ↓ 循环
```

### 4.3 Builder 对 Builder（开发者）的核心价值

一个 Builder 在平台上应该能获得：

1. **效率**：AI Build 10 分钟完成过去 2 小时的方案设计
2. **复用**：一套方案可以部署到 100 个客户
3. **收入**：发布方案模板到市场获得收入分成
4. **协作**：团队协作完成大型项目交付
5. **运维**：远程监控和诊断已交付项目
6. **成长**：从 Free Builder 成长为认证服务商

## 五、与 Studio 的集成路径

### 5.1 当前

- Builder 创建项目 → 手动关联 Workspace 内 Studio
- 方案「Open in Studio」= 在 Builder 内生成新项目

### 5.2 未来

| 阶段 | 能力 |
|------|------|
| **Phase 1** | Builder 方案 → 一键下发到 Studio（自动化规则、设备配置） |
| **Phase 2** | Studio 运行数据 → 实时回流到 Builder 项目面板 |
| **Phase 3** | Builder 远程编排 → 通过 Studio Cloud 下发到 Studio + 网关 |
| **Phase 4** | Studio Space Ontology → Builder 可查询、可视化、可分析 |

### 5.3 数据流

```
Builder (设计态)                    Studio (运行态)
    │                                   │
    │  方案下发（自动化、配置、界面）      │
    │ ─────────────────────────────────→ │
    │                                   │
    │  数据回流（状态、事件、服务结果）    │
    │ ←───────────────────────────────── │
    │                                   │
    │  远程编排（Actions、Skills、更新）  │
    │ ─────────────────────────────────→ │
```

## 六、Builder 的成长阶段

| 阶段 | Builder 平台状态 | 用户价值 |
|------|-----------------|---------|
| **MVP（当前）** | 创作 + 方案 + 项目 + 基础空间设计 + 选品 | 能用 AI 快速设计方案，管理项目 |
| **V1.0** | + 高质量 3D 预览 + AI 自动化 + 方案模板市场 + Studio 关联 | 方案可复用、可交易，项目可关联 Studio |
| **V1.5** | + 方案下发到 Studio + 运维数据回流 + 远程诊断 | 设计到交付的完整闭环 |
| **V2.0** | + 开发者 SDK + 插件机制 + Skill 市场 + 行业模板 | 生态繁荣，第三方在 Builder 上创造价值 |
| **V3.0** | + Space Ontology 可视化 + Agent 编排 + 多站点批量管理 | 企业级空间智能控制面 |

## 七、结论

Builder 的终极目标不是一个网页工具，而是**空间智能的云端控制面和生态面**。

- **短期**：让 Builder 用 AI Build 10 分钟完成一个方案，管理一个项目
- **中期**：让方案可复用、可交易，项目可关联 Studio 闭环交付
- **长期**：让 Builder 成为 Aqara 空间操作栈的上层应用入口，开发者生态在此繁荣

**验证标准**：一个 Free Builder 注册后，能否在 30 分钟内完成「创作方案 → 3D 预览 → 生成项目 → 选品采购」的完整体验？如果能，Builder 的核心价值就立住了。
