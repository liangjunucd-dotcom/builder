## 执行摘要

### 一个根本性的判断

Aqara 过去靠门店和服务商卖货，覆盖有限。Builder 的出现，是为了让全球的创作者和集成商帮 Aqara 覆盖它无法直接触达的每一个角落——**共创、共享、共利**，这是 Builder 存在的商业本质。

但在 AI Agent 时代，这件事有了第二层意义：

AI 会让"创作"越来越便宜，但它无法替代两件事——**让生成的内容真正跑进物理空间**，以及**沉淀对真实空间的结构化理解**。

这两件事，正是 Builder 生态能做、也必须做的事。

### 核心结论

**Aqara Builder 的终极定位是：空间智能的世界模型沉淀网络与共利基础设施。**

它不是开发者社区，不是插件商店，不是模板市场。它是一个让全球 Builder 把对空间的理解沉淀成可执行资产、可被 AI 调用、可被真实部署、可以持续变现的平台。

Builder 赚到钱，Aqara 卖出硬件，用户得到更好的空间体验——三方利益在同一个机制里自然对齐。

### 关键行动优先级

| 优先级 | 行动 | 时间 | 战略意义 |
|--------|------|------|---------|
| 立即 | 让空间包真正可部署（Builder→Studio 打通） | 0-6月 | 没有部署闭环，一切都是展示 |
| 立即 | 灵感广场变获客入口（需求线索机制） | 0-6月 | 让 Builder 马上能赚钱 |
| 短期 | 能力资产的标准化描述（机器可读） | 6-12月 | AI Agent 调用的前提 |
| 短期 | Builder 认证与收益体系 V1 | 6-12月 | 供给侧动力的制度保障 |
| 中期 | Agent 调用计费体系 | 12-24月 | 第二收入曲线 |
| 中期 | 空间本体开放标准（ORAP） | 12-24月 | 行业话语权与护城河 |

## 第一章：Builder 是什么，以及为什么现在做不够

### 1.1 为什么做 Builder

Aqara 的门店模式有天然上限——一家门店只能覆盖一个城市的一部分客户，而全球有数十亿个需要被智能化的空间。

Builder 是 Aqara 破局的方式：**把"销售和服务"这件事，从自己做变成生态做**。全球的集成商、开发者、设计师，他们最了解本地市场、本地用户、本地建筑习惯。让他们用 Aqara 的平台构建解决方案，卖给他们的客户，Aqara 收硬件收入和平台分成，Builder 收服务费和内容收入。

这不是新概念——Shopify 是这样做的，Salesforce AppExchange 是这样做的，App Store 是这样做的。Aqara 做的是物理空间版本的这件事。

### 1.2 当前 Builder 的真实问题

现在 Builder 平台有三个根本性的断裂：

**第一个断裂：创作和部署之间的断裂。**
Builder 里 AI 生成了方案，但这个方案无法直接下发到 Studio 运行。创作出来的东西，落不了地。这让"灵感广场"变成了一个作品集展示区，而不是一个能产生真实价值的市场。

**第二个断裂：创作者和客户之间的断裂。**
Builder 在平台上发布了方案，但没有机制让有需求的客户找到这个 Builder。方案发出去之后，没有后续，没有线索，没有收入。Builder 没有留下来的经济动机。

**第三个断裂：内容和价值之间的断裂。**
社区里的帖子、模板、教程，没有清晰的变现路径，也没有与 Builder 的专业身份和商业价值挂钩。发帖是发帖，赚钱是赚钱，两件事没有连接。

**这三个断裂不解决，社区永远热闹不起来。** 因为人们聚集在有价值流动的地方，而不是有内容展示的地方。

### 1.3 AI Agent 带来的第二层机会

在解决上述基础问题的同时，AI 的发展给 Builder 带来了一个更大的战略机会——但这个机会不是"用 AI 替代 Builder"，恰恰相反。

AI 会越来越擅长生成方案，但它面临一个无法绕过的问题：**它不知道真实空间里有什么，也无法保证在真实空间里稳定运行。**

要让 AI Agent 真正能操作物理空间，它需要：
- 知道这个空间里有哪些设备、它们是什么关系、支持什么操作（**空间语义**）
- 能通过标准化接口调用这些能力（**能力 API**）
- 有机制保证调用是安全的、有权限的（**治理层**）
- 有人在物理世界负责安装、调试、运维（**Installer 网络**）

这四件事，正是 Builder 生态应该沉淀和提供的东西。**Builder 不是被 AI 替代的对象，而是 AI 操作物理世界的必经之路。**

## 第二章：重新定义 Builder

### 2.1 一句话定位

> **Aqara Builder 是空间智能的世界模型沉淀网络——让全球 Builder 把对空间的理解变成可执行资产、可被 AI 调用、可被真实部署、可以持续变现。**

这个定位的四个关键词：
- **世界模型**：不只是功能，而是对空间的结构化理解
- **可执行**：不只是看，而是能真正部署到 Studio 运行
- **可被 AI 调用**：为 Agent 时代预留接口，这是未来的增量价值
- **可以变现**：这是 Builder 留在平台的根本动力

### 2.2 Builder 的长期使命

**使命一：让空间经验可以流通。**
一个有十年经验的养老院改造集成商，他脑子里的知识今天只能服务他本地的几十个项目。Builder 让他把这些知识打包成空间方案，流通到全球每一个有同样需求的角落。

**使命二：让物理空间对 AI 可见。**
AI 要真正能操作物理空间，首先要理解物理空间。Builder 生态沉淀的空间本体和能力描述，是 AI 认知物理世界的基础数据。

**使命三：让更多人参与空间智能的创造。**
不只是 Aqara 内部的工程师，全球每一个懂空间的人——建筑师、物业管理员、养老服务商、宠物店主——都应该能把他们的理解变成可运行的空间智能产品。

**使命四：让 Builder 因为贡献而获得真实收益。**
平台存在的正当性，建立在每个参与者都能从中获益上。Builder 卖货、赚服务费、获得被动收入，才是这个生态能持续运转的基础。

### 2.3 核心价值主张

**对集成商和服务商：**
你过去卖的是时间。做完一个项目，这个项目的经验就锁在你脑子里。现在你可以把项目经验打包成空间方案，发布到平台，被全球其他项目复用。你的第十个项目产生的收入，会在你做第二十个项目时还在流入。

**对开发者：**
物理世界现在有 API 了。`motion_sense`、`door_window_sense`、`humidity_control`——这些设备能力以标准化接口暴露出来，你用 Web 技术就能构建空间应用。不需要学习 Zigbee 协议，不需要了解硬件，你已经有了 80% 需要的技能。

**对行业专家（不写代码的人）：**
你最了解养老院里老人的需求，最了解酒店客房服务的流程，最了解水培植物的生长规律。这些行业知识今天变成不了产品。Builder 的 AI Build 工具让你用描述自己需求的方式，就能构建出可以运行的空间方案。

**对 Aqara：**
每一个 Builder 都是一个不需要 Aqara 管理的销售渠道。Builder 帮客户落地方案，需要购买 Aqara 的 Studio 和设备，Aqara 收硬件收入；Builder 在平台上销售内容，Aqara 收平台分成；AI Agent 调用平台上的能力，Aqara 收调用费。这是一个三层叠加、随生态规模自动增长的收入结构。

## 第三章：Builder 应该沉淀什么

这是最容易做错的地方。很多平台沉淀的是"内容"——帖子、评论、截图——这些东西对平台没有真实的战略价值。

Builder 应该沉淀的，是**有执行价值的结构化资产**，而不是有传播价值的非结构化内容。

### 3.1 五层内容资产体系

**第一层：空间世界模型（最高价值，最难沉淀）**

这是对一类空间的结构化描述——它有哪些对象，对象之间什么关系，哪些行为在这里成立，哪些约束必须遵守。

例如：「两居室出租公寓的世界模型」描述了：这类空间通常有哪些房间，每个房间里有哪些功能对象，租客的权限边界在哪里，入住和退租的标准流程是什么。

这层内容是 AI 理解和操作一类空间的基础。它的价值不在于单次使用，而在于被越来越多的方案引用和优化。它是真正的平台资产，不属于任何单个 Builder。

沉淀方式：从大量空间包中自动提取共性结构，经专家审核后沉淀为标准世界模型。

**第二层：空间方案包（核心交易资产）**

这是针对具体场景的完整解决方案——包含空间结构、设备需求声明、自动化逻辑、用户界面定义、参数配置和演示数据。

关键要求：方案包必须是**可执行的**，不是截图，不是文字描述。下发到 Studio 后能直接运行，经过现场设备映射后立刻工作。

这是 Builder 平台最核心的交易商品。Builder 销售方案包，Aqara 抽取佣金，客户购买后带动硬件销售——三方利益在这个商品上直接对齐。

**第三层：能力资产（可被 AI 调用的标准组件）**

这是比方案包更细粒度的组件——一个经过测试和认证的自动化逻辑片段、一个设备能力的标准封装、一个特定场景的处理规则。

关键要求：每个能力资产必须有机器可读的描述规范（Tool Schema）——它能做什么、需要什么权限、输入输出是什么、可靠性如何。这是 AI Agent 能调用它的前提。

**第四层：行业知识资产（软性但重要）**

这是 Builder 在真实项目中积累的经验——某类建筑的施工特点、某个行业的合规要求、某类用户的使用习惯、某种设备在特定环境下的表现。

这层内容的价值不是直接可执行，而是帮助其他 Builder 更快做出更好的方案。它是平台知识库的核心，也是认证体系和声誉体系的基础。

**第五层：Builder 身份资产（平台网络资产）**

这是 Builder 在平台上积累的专业身份——发布了哪些方案、服务了哪些项目、获得了什么认证、在哪些行业被同行认可。

这层资产的价值在于，它让 Builder 不只是平台的用户，而是平台网络的节点。一个有足够身份资产的 Builder，他的方案会在推荐算法中获得更高的优先级，他的服务报价可以更高，他能接到更多高质量的项目线索。

### 3.2 内容沉淀的时间线

| 阶段 | 重点沉淀层 | 原因 |
|------|-----------|------|
| 现在-12月 | 方案包（可执行） + Builder 身份资产 | 先让基础交易跑通，先让 Builder 赚到钱 |
| 12-24月 | 能力资产（标准化） + 行业知识 | 生态有足够供给后，开始标准化 |
| 24月以上 | 世界模型（从方案中提取） | 需要足够的数量才能提取有价值的共性结构 |

## 第四章：AI Build 的重新定义

### 4.1 AI Build 不是"降低门槛的玩具"

当前 AI Build 的问题：它生成了方案，但方案和 Studio 之间有断层，没有部署路径。这让 AI Build 变成了一个展示功能，而不是一个生产力工具。

AI Build 应该成为的东西：**一个从意图到可部署方案的完整生产工具链。**

### 4.2 AI Build 产出的最小单元：空间方案包

一个完整的空间方案包包含六层：

```
空间方案包
├── 意图声明（这个方案要解决什么问题）
├── 空间结构（适用于什么类型的空间，有哪些区域）
├── 能力槽位（需要哪些类型的设备能力，不指定品牌型号）
├── 逻辑规则（在什么条件下执行什么操作）
├── 用户界面（终端用户在 Life App 里看到什么）
└── 部署配置（需要什么 Studio 版本，有哪些前置条件）
```

关键：**能力槽位解耦了方案和具体设备**。一个"湿度管理"方案里的"湿度感知"槽位，可以被任何通过 Aqara 认证的温湿度传感器填充。方案在不同品牌、不同型号的设备上都能运行。

### 4.3 方案包的生命周期

```
AI 辅助创建  →  私有调试  →  模拟器验证  →  发布（公开/受限/团队）
      ↓                                            ↓
用户安装配置  ←  设备映射  ←  下发 Studio  ←  购买或订阅
      ↓
真实运行  →  数据反馈  →  方案迭代
```

每一个环节都需要真实的产品能力支撑，不能有断层。

### 4.4 方案包的分享层级

| 层级 | 适用场景 | 商业化方式 |
|------|---------|-----------|
| 私有 | Builder 的私有方案库，用于自己的项目 | 无 |
| 团队共享 | 公司内部或合作团队之间共享 | 团队订阅 |
| 受限分享 | 指定购买者或受邀者才能访问 | 按份付费或订阅 |
| 公开免费 | 社区共享，吸引流量和声誉 | 间接价值（获客、身份） |
| 公开付费 | 在平台市场销售 | 直接收益，平台抽成 20% |

## 第五章：让 Builder 赚到钱——五条收入流的机制设计

这是整个平台能否运转的核心。**Builder 赚不到钱，任何社区设计都没有意义。**

### 5.1 收入流一：项目服务费（最快能实现，现在就应该有）

Builder 在平台上展示自己的方案和能力，有项目需求的客户找到他，双方谈合作，Builder 赚取项目服务费。

**机制设计：**
- 灵感广场每个方案下增加"我想在我的空间实现这个"按钮
- 用户填写需求（空间类型、面积、预算、城市），提交后转化为客户线索
- 线索推送给方案创建者和附近的认证 Builder
- Builder 主动联系客户，谈合作

这一个改动，就把灵感广场从"作品展示"变成了"获客漏斗"。Builder 发布方案的动机从"秀作品"变成"获客户"，发布质量和频率都会提升。

### 5.2 收入流二：方案包销售（被动收入，规模化的关键）

Builder 把经过项目验证的方案打包，在平台市场销售。其他 Builder 或直接用户购买，用于自己的项目或空间。

**关键设计：**
- Builder 获得 80%，Aqara 平台抽成 20%
- 方案包绑定设备推荐（BOM），每通过方案包带动的硬件销售，Builder 额外获得 8-12% 的硬件佣金
- 方案包被其他 Builder Fork 和改进时，原作者持续获得 Remix 分成

这让 Builder 从"卖时间"升级到"卖资产"——一个好的方案包可以持续产生收入，而不是项目结束就停了。

### 5.3 收入流三：Mini App 订阅（面向终端用户的内容变现）

Builder 在 Life App 里发布垂直场景 App（如 MyGuard 安全守护、是否活着老人关怀），终端用户订阅或一次性购买。

Builder 获得 80%，Aqara 抽成 20%。

这条收入流的特点是：一次开发，全球用户，边际成本接近零。一个做得好的 Mini App，可以持续为 Builder 带来被动收入。

### 5.4 收入流四：能力调用费（AI Agent 时代的增量收入）

当 AI Agent 通过 Builder 平台调用 Builder 创建的能力时，每次调用产生计费，Builder 获得收益。

这条收入流现在还很小，但它是未来三到五年内增长最快的部分。Builder 今天创建并发布到平台的能力资产，在 Agent 时代会持续产生调用收益——即使 Builder 本人不再主动维护，资产还在工作。

### 5.5 收入流五：持续运维服务（高黏性、经常性收入）

Builder 为企业客户提供月度或年度的系统维护服务——空间系统优化、设备升级、故障响应、功能迭代。

这条收入流让 Builder 和客户建立长期关系，而不是一次性交付。对 Builder 来说，它是经常性收入；对客户来说，它是服务质量的保障。

### 5.6 收入流的时间线

```
现在   →  项目服务费（立刻能做）
          ↓
6月    →  方案包销售（需要 Builder-Studio 打通）
          ↓  
12月   →  Mini App 订阅（需要 Life App 应用市场）
          ↓
24月   →  能力调用费（需要 Agent API + 计费体系）
          ↓
持续   →  运维服务（随客户基础自然增长）
```

## 第六章：用户旅程——从陌生人到核心 Builder

### 阶段零：发现（进来的理由）

**用户目标：** 找灵感，看看有没有解决我问题的方案

**产品设计重点：**
- 搜索和发现体验要基于"问题"而不是"产品类别"——用户搜索"老人独居安全"找到方案，而不是搜索"传感器自动化"
- 每个方案展示：解决什么问题 + 需要哪些设备 + 已有多少人在用 + 联系创作者的入口
- 设备匹配提示：系统检测用户已有设备，高亮"你的设备可以支持这个方案"

**平台沉淀：** 用户的搜索意图和浏览行为，这是供给侧的需求信号

---

### 阶段一：安装（第一次价值体验）

**用户目标：** 把一个方案装进我的空间，看看效果

**产品设计重点：**
- 安装向导：自动检测已有设备 → 展示匹配的槽位 → 引导填入设备 → 下发 Studio → 激活
- 整个过程不超过 10 分钟，不需要懂技术
- 缺少设备时，直接推荐购买（附购买链接），计入方案创建者的硬件佣金

**平台沉淀：** 哪些方案被真实安装，设备槽位的满足率，这是衡量方案质量的真实数据

---

### 阶段二：创作（从消费者到创作者）

**用户目标：** 定制一个适合我的方案，或者创作一个解决特定问题的方案

**产品设计重点：**
- 创作入口不只是"从空白开始"，更多是"基于已有方案修改"——Fork 一个社区方案，调整参数，存为自己的版本
- AI Build：描述你的需求，AI 生成方案草稿，用户在可视化界面中调整
- 模拟器：在没有真实设备的情况下，用虚拟设备测试逻辑

**平台沉淀：** Builder 的创作行为记录，Fork 关系图谱，这构成了内容的进化脉络

---

### 阶段三：发布（从创作者到供给方）

**用户目标：** 把我创作的方案分享出去，或者在市场上销售

**产品设计重点：**
- 发布流程：填写方案信息 → 选择分享层级（私有/受限/公开/付费）→ 提交审核 → 上线
- 付费方案需要通过更严格的审核（设备兼容性测试 + 逻辑安全审查）
- 发布后：实时看到浏览量、安装量、收益数据

**平台沉淀：** 方案库的内容供给，Builder 的发布历史和收益记录

---

### 阶段四：交付（从创作者到服务商）

**用户目标：** 为客户交付一个完整的空间项目，赚取服务费

**产品设计重点：**
- 项目工作区：Builder 为每个客户项目创建独立工作区，管理方案定制、设备清单、部署状态、验收记录
- 需求市场：平台汇总来自灵感广场的客户线索，推送给匹配的 Builder
- 协作工具：邀请客户进入工作区，共同确认方案，实时看到部署进度

**平台沉淀：** 交付记录、客户评价、项目案例，这是 Builder 专业身份的核心证明

---

### 阶段五：沉淀（从服务商到行业专家）

**用户目标：** 把项目经验转化为平台资产，建立行业声誉，形成被动收入

**产品设计重点：**
- 项目完成后，一键将项目中的定制方案发布为新的市场商品
- 经验提炼工具：引导 Builder 记录项目中的关键决策和踩过的坑，这些记录沉淀为行业知识库
- 声誉仪表盘：实时展示 Builder 的方案被安装次数、客户评分、收益积累、行业排名

**平台沉淀：** 行业知识库的内容，Builder 的声誉数据，这是平台最难被复制的资产

## 第七章：留存机制——Builder 为什么不走

### 7.1 五种留存的本质

**资产留存（最强）：** Builder 在平台上发布的方案、积累的客户评价、建立的声誉等级——这些东西带不走。他离开 Builder 平台，就失去了这些资产。

**收益留存（最直接）：** 如果 Builder 每个月因为方案销售和调用费收到可观的被动收入，他会有很强的动机留在平台。这要求平台的变现机制真实可用，而不是理论上存在。

**关系留存：** Builder 与客户的长期运维关系，与其他 Builder 的合作关系，都沉淀在平台上。客户习惯了通过平台的工作区与他沟通，关系迁移成本高。

**项目留存：** 所有进行中的项目、历史项目的记录、与客户的沟通历史，都在平台工作区里。迁移一个有十个活跃项目的 Builder，他需要重建所有的项目管理工具和客户关系。

**身份留存：** Builder 的认证徽章、在社区的贡献积分、被引用最多的方案——这些构成了他在这个行业里的专业身份。在其他地方，他是从零开始的。

### 7.2 AI Agent 带来的新型留存

随着 AI Agent 开始调用 Builder 平台上的能力资产，会产生一种新的留存动力：**被动调用收益**。

Builder 发布的方案和能力资产，每次被 Agent 调用都会产生收益。这些收益不需要 Builder 做任何额外工作。这种被动收益越积累，Builder 就越不会离开——因为他的资产在帮他赚钱。

## 第八章：成长与认证体系

### 8.1 成长等级

不用太复杂，五级就够，每级有清晰的升级条件和对应权益：

| 等级 | 名称 | 条件 | 核心权益 |
|------|------|------|---------|
| L1 | Explorer | 注册 + 完成入门认证 | 基础创作工具，社区访问 |
| L2 | Builder | 发布 3 个方案 + 订阅年费 | 市场销售资格，硬件佣金 8% |
| L3 | Pro Builder | 年方案销售额 $10k+ 或 10+ 项目交付 | 优先审核，佣金 12%，需求线索优先推送 |
| L4 | Expert Builder | 特定行业认证 + 年销售额 $50k+ | 官方推荐资源，佣金 15%，行业标杆资格 |
| L5 | Partner | Aqara 邀请制，战略合作 | 联合营销，OEM 资格，共同产品定义权 |

### 8.2 专业认证体系

除了等级，Builder 还可以获得垂直行业的专业认证：

- **住宅认证 Builder**：完成住宅标准课程 + 5 个验收项目
- **酒店认证 Builder**：完成酒店专项课程 + 合规要求培训 + 3 个酒店项目
- **养老认证 Builder**：完成养老专项课程 + 2 个养老机构项目
- **办公认证 Builder**：完成办公专项课程 + 3 个企业项目

认证的价值是双向的：对 Builder，认证是市场信誉的证明；对 Aqara，认证确保了每个行业赛道上有足够质量的 Builder 覆盖。

### 8.3 贡献值系统

贡献值是让社区健康运转的激励机制，但必须和真实价值挂钩，而不是"点赞数"：

| 行为 | 贡献值 | 原因 |
|------|--------|------|
| 方案被安装 1 次 | +10 | 真实价值的体现 |
| 方案被付费购买 1 次 | +50 | 商业价值更高 |
| 项目交付 + 客户五星好评 | +200 | 最高质量的贡献 |
| 知识文章被收藏 | +5 | 软性价值 |
| 方案被 Fork 后再发布 | +30 | 知识传播 |
| Agent 调用我的能力 100 次 | +100 | AI 时代的新贡献形式 |

贡献值影响：需求线索的推送优先级、在搜索结果中的排名、解锁高级功能的门槛。

## 第九章：共创共享共利的机制设计

### 9.1 共创：不是一起写一个方案，而是站在对方肩膀上

真正的共创不是多人同时编辑一个文档，而是**方案可以进化**——A 创建了一个基础方案，B 在上面改进后发布，C 在 B 的基础上再优化，每个版本都记录来源，每次 Fork 都让原作者获益。

**技术机制：**
- 每个方案包有版本树，Fork 关系可视化
- Fork 发布需要原作者授权（可设置"开放 Fork"或"受限 Fork"）
- Fork 方案产生的收益，按设定比例（如 10-30%）自动分配给原作者

**团队工作区：**
- Builder 可以创建团队工作区，邀请成员共同创作
- 工作区内支持方案的协同编辑、讨论、版本管理
- 工作区产出可以以团队名义发布，收益按贡献度分配

### 9.2 共享：不是所有内容都应该公开

共享的精髓是**让正确的内容流向正确的人**，而不是一切公开或一切封闭。

前面的五层内容分级已经定义了共享层级，这里补充关键原则：

**基础共享是平台活力的来源：** 行业知识、基础方案模板、教程内容，应该鼓励公开共享，这是新 Builder 进来的第一个价值。

**能力资产的共享要有商业化路径：** 精心打磨的方案包和 Mini App，Builder 有权选择付费模式。平台不应该强制要求高价值内容免费。

**世界模型层的内容应该走向开放：** 对特定行业空间的通用描述（如"两居室出租公寓的标准模型"），应该逐步开放给整个生态，包括第三方 AI 调用，这是 Aqara 构建行业标准地位的方式。

### 9.3 共利：利益绑定是生态持续运转的基础

最重要的设计原则：**平台的收益应该和 Builder 的贡献直接挂钩，而不是平台自己获益、Builder 做贡献。**

具体机制设计：
- Aqara 不与 Builder 竞争——Aqara 做平台能力和基础方案模板，不做垂直行业的商业解决方案
- 硬件佣金制度——Builder 带动的硬件销售，Builder 获得佣金（8-15% 按等级）
- 平台抽成阶梯式——等级越高，抽成比例越低（20% → 15% → 10%）
- Agent 调用分成——能力资产被 AI Agent 调用，Builder 按次获益
- 早期贡献者超额回报——前 200 个认证 Builder 享受终身最低抽成率，成为平台的传教士

## 第十章：Builder 与 Studio 的关系

这个关系需要说清楚，否则产品团队会持续产生边界混乱。

### 10.1 核心原则

**Studio 是运行时，Builder 是创作和交易层。** Studio 的东西在本地跑，Builder 的东西在云上沉淀。两者服务不同的用户，解决不同的问题，但紧密协作。

### 10.2 清晰的边界

| 维度 | Studio | Builder |
|------|--------|---------|
| 主要用户 | 设备安装者、专业管理员 | 方案创作者、集成商、开发者 |
| 核心价值 | 可靠运行、本地控制、物理执行 | 创作、交易、社区、沉淀 |
| 数据存在 | 本地（断网可用） | 云端（可同步） |
| 界面形态 | 专业管理控制台（Studio Web） | 创作和交易平台（网页 + App） |
| AI 角色 | AI 在本地推理和执行 | AI 辅助创作，AI 调用能力 |

### 10.3 协作机制

**Builder → Studio：** Builder 上创作的方案包，通过 Studio Cloud 下发到本地 Studio。Studio 接收方案，等待现场设备映射，映射完成后开始运行。

**Studio → Builder：** Studio 运行中积累的匿名化运行数据（哪些规则触发频繁、哪些设备离线率高），反馈到 Builder 平台，帮助创作者优化方案。

**Builder 不是 Studio 的远程管理界面。** 远程管理是 Studio Web 的职责（通过 Studio Cloud 代理）。Builder 是在 Studio 之上的独立创作和交易层。

## 第十一章：面向 AI Agent 的技术准备

这一章是为未来 18-24 个月布局的。现在不做，未来会付出更高的代价。

### 11.1 能力资产的机器可读规范

Builder 平台上的每一个能力资产，必须有机器可读的描述，让 AI Agent 能理解和调用它：

```yaml
capability:
  id: "aqara.skill.elderly-checkin-monitor"
  name: "老人活动签到监测"
  
  description:
    human: "检测老人是否在设定时间窗口内有正常活动"
    machine: "时间窗口内运动传感器无触发则触发通知"
  
  required_slots:
    - capability_type: "motion_sense"
      min: 1
      max: 5
      
  user_params:
    - name: "checkin_window_start"
      type: "time"
      default: "08:00"
    - name: "notification_recipients"
      type: "user_list"
      
  permissions:
    required: ["device.sensor.read", "notification.send"]
    
  reliability: "99.5%"
  
  pricing:
    invoke: 2  # Credits per check
    
  certification: "verified"
```

### 11.2 Agent 调用的入口设计

为 Agent 提供标准化的能力注册中心入口：

- Agent 可以查询"这个空间里有哪些可用能力"
- Agent 可以按照 Tool Schema 调用具体能力
- 每次调用经过权限验证和计费处理
- 调用结果包含执行确认（设备状态实际变化的验证）

### 11.3 空间本体开放标准（ORAP）

Objects（对象）、Relations（关系）、Actions（动作）、Policies（策略）——这是 Aqara 正在构建的空间描述标准。

战略建议：**把 ORAP 作为开放标准提交给行业协会，邀请竞争对手和第三方参与标准建设。**

这看起来是"放弃竞争优势"，实际上恰恰相反——标准的制定者是最大的受益者。当 ORAP 成为行业标准，所有基于这个标准构建的 AI Agent 都需要依赖能最好支持这个标准的平台，而 Aqara 是最早、最深度实现这个标准的。

这是 Android 策略——通过开放赢得生态，而不是通过封闭保护自己。

## 第十三章：最容易陷入的误区

**误区一：把热闹当目标。**
论坛里有人发帖不等于平台有价值。真正的活跃是有价值交换的活跃——方案被购买、需求被满足、Builder 赚到钱。没有这些，表面的热闹是虚假繁荣。

**误区二：Aqara 自己做方案和 Builder 竞争。**
Aqara 做了一个非常好的"住宅安防方案"，那所有做安防的 Builder 就把 Aqara 当竞争对手。Aqara 的角色是平台，不是参赛者。官方只做"基础参考模板"，不做商业化的垂直解决方案。

**误区三：等平台完善再宣传。**
没有完美的开始。前 200 个 Builder 是最重要的——他们会成为平台的传教士，帮 Aqara 招募接下来的 2000 个。现在就应该给早期加入的人超额回报，而不是等一切都好了才开放。

**误区四：只做工具，不做治理。**
随着生态打开，会出现低质量内容、虚假评价、抄袭纠纷。没有治理体系，平台的信任会快速崩溃。审核机制、评价体系、争议处理，必须和内容市场同步上线，而不是事后补救。

**误区五：认为 AI 会把 Builder 变得不必要。**
恰恰相反。AI 让"生成内容"变得便宜，但让"可信任的、可执行的、有真实世界根基的内容"变得更稀缺。Builder 生态的价值不在于生成内容，而在于保证内容的质量、可执行性和真实世界的可靠运行。
