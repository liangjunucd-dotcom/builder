## 前言：用户说了什么，真正意味着什么

> *"如果只是自动化就还好，但感觉壁垒不是那么强，也没那么吸引用户。他们想知道户型图下面，哪些地方用到设备，安装在哪里，有个自动生成平面或 3D 的能力不错。"*

这个反馈背后有一个深刻的产品洞察：

**自动化是"逻辑"，而用户购买的是"空间体验"。**

建筑师画施工图，客户看的不是结构计算公式，而是效果图里那个自己未来生活的空间。  
同样，智能空间方案的核心呈现，不应该是"规则列表"，而应该是：

> **"这是你家的平面图，这里是主卧的存在感应器，这里是玄关的门锁，橙色的灯光路径就是当你回家时，整套方案触发的联动顺序。"**

这一个转变，将 Aqara Builder 从"配置工具"跃升为"空间智能设计平台"。

## 第一章：重新定义产品层次——六层空间智能栈

当前 v1 版本的 XML(Builder) 覆盖了中间三层。v2 的战略方向是打通全六层。

```
┌──────────────────────────────────────────────────────────────────┐
│  Layer 6  │  空间可视化 & 方案呈现                                │
│           │  设备落位平面图 · 3D 场景漫游 · 可交付给业主的图纸    │
├──────────────────────────────────────────────────────────────────┤
│  Layer 5  │  App UI 资产层                                        │
│           │  Asset 卡片 · 场景入口 · 状态仪表盘                   │
├──────────────────────────────────────────────────────────────────┤
│  Layer 4  │  自动化逻辑层                                         │
│           │  原子逻辑块 · 触发/条件/动作 · 跨协议能力点           │
├──────────────────────────────────────────────────────────────────┤
│  Layer 3  │  设备实例层（虚拟 → 真实绑定）                        │
│           │  Aqara 设备 + BACnet/Modbus 数据点 + Matter 桥接      │
├──────────────────────────────────────────────────────────────────┤
│  Layer 2  │  空间语义层                                           │
│           │  空间图谱 · 房间功能语义 · 区域关系                   │
├──────────────────────────────────────────────────────────────────┤
│  Layer 1  │  物理空间层（输入层）                                  │
│           │  户型图 / CAD / BIM / 拍照扫描                        │
└──────────────────────────────────────────────────────────────────┘
```

**v1 强项：** Layer 2-5 均有 XML(Builder) 支撑，自动化和 App UI 已具备。  
**v2 补足：** Layer 1（物理空间输入）和 Layer 6（可视化呈现输出），是用户感知最强、差异化最大的两端。

**六层栈与产品载体的统一映射（避免「战略层」与「产品名」脱节）：**

| 能力范围 | 主要载体 | 说明 |
|---|---|---|
| Layer 1～2 输入与设计语义 | **Aqara Builder**（**Workspace** 为租户/计费/协作边界） | 户型/描述 → XML(Builder)；项目与云资源挂在 Workspace 下 |
| Layer 3～5 运行与绑定 | **Studio Core**（M300 / **Cloud Studio**） | 知识图谱、设备网关、自动化执行；**同一 Runtime** |
| 工程实施 GUI | **Studio Web** | 绑定、组态大屏、XML(Builder) 导入导出；面向交付与运维 |
| Layer 5～6 日常呈现 | **Aqara Life**（插件引擎） | 插件消费 XML(Builder)/语义 ID；**底部「生活·市场·商城·我」**见第十二章 |
| 纳管 / 分发 / 中继 / 推送通道 | **Studio Cloud**（**按 Workspace 绑定区域实例**，如 US/EU/CN） | **不做业务状态主存**；区域由 Workspace **数据中心（Data Region）**决定 |

## 第二章：Layer 6——空间可视化，用户真正愿意付费的东西

### 2.1 用户要的不是"规则"，是"设计图"

看附件截图："建安 · 照明设计 丨 天花"。这是一位方案设计师输出的灯光方案，给业主确认的。  
这张图让业主**看得懂、能判断、愿意签字**。

Aqara Builder 生成的 XML(Builder)，应该能自动产出等价于这张图的智能化施工图：

```
┌──────────────────────────────────────────────────────────┐
│  Aqara Builder 可视化输出物                               │
│                                                          │
│  ① 设备落位平面图                                        │
│     • 每个房间的平面轮廓 + 家具摆放（导入或 AI 生成）    │
│     • 设备图标按真实位置标注（传感器/灯具/开关/门锁）    │
│     • 连线显示联动关系（触发设备 → 受控设备 用彩色箭头）  │
│     • 输出为可打印 PDF / 可共享链接                      │
│                                                          │
│  ② 自动化逻辑图                                          │
│     • 场景触发路径可视化（人回家 → 哪些设备按什么顺序响应）│
│     • 安全告警路径图（烟雾 → 通知链 + 联动动作）         │
│                                                          │
│  ③ 3D 方案漫游（增强版，后期）                           │
│     • 基于空间语义生成简单 3D 场景                       │
│     • 可切换"日间 / 夜间 / 回家 / 离家"模式展示灯光效果  │
└──────────────────────────────────────────────────────────┘
```

### 2.2 为什么这是护城河

1. **生成设备落位图需要设备知识**：传感器应该装在天花板哪个位置、开关面板在哪面墙、门磁对应哪扇门——这些规则来自 Aqara 多年设备安装经验。竞争者没有这个知识库。

2. **可视化输出提升了方案的"可交付性"**：方案商给业主展示的不再是截图，而是专业的智能化设计图。这使方案商愿意为工具付费（因为工具帮他们赚钱）。

3. **图纸 = 施工依据**：安装商从设备落位图上明确知道每个设备装在哪里，显著减少现场沟通成本和返工。

4. **视觉资产进入传播链**：漂亮的方案图会被方案商发到朋友圈、小红书、行业群，带来自然流量。

## 第三章：商业护城河分析——六层壁垒

用户反馈说"自动化壁垒不强"——是对的。自动化是可以被复制的。但以下六层壁垒，是累积而成、无法被快速复制的。

### 壁垒一：硬件生态深度（当下最强）

Aqara 是苹果生态（HomeKit / Matter）覆盖最全的设备供应商。  
在 Apple Watch、Siri、HomeKit 场景下，Aqara 的设备密度无可比拟。  
**Builder 的设备推荐、BOM 估算、安装规范，都内化了这套设备知识**——这不是一个 API 可以接入的东西，是多年产品积累。

### 壁垒二：空间智能设计专有知识（中期核心）

```
设备落位知识图谱（专有）
│
├── 传感器安装规范（FP2 最佳覆盖角度、高度、避免遮挡）
├── 布线建议（哪些设备需要预留强电，哪些纯无线）
├── 空间功能 → 设备推荐映射（已在 device-catalog 中部分实现）
├── 安全规范（厨房设备安全距离、燃气传感器位置规范）
└── 照明设计规范（色温分区、亮度梯度、灯带安装位置）
```

这些知识编译进 XML(Builder) 生成逻辑后，形成了**比任何通用 AI 都更懂智能家居安装**的专域模型。

### 壁垒三：XML(Builder) 方案语料库（长期最强）

```
XML(Builder) 方案语料的价值密度
│
├── 空间语义 × 设备组合（"三室两厅 + 这组设备"是否合理）
├── 自动化逻辑 × 实际效果（"这条规则"在真实空间里跑了多久）
├── Remix 轨迹（"这个方案被改了什么"反映了用户偏好）
└── 部署反馈（"这个方案在哪类空间部署成功率高"）
```

每一份真实部署的 XML(Builder)，都是**结构化、有标注、可用于训练智能空间 AI 的数据**。  
这个语料库的稀缺性在于：它不是从互联网爬来的文字，而是**经过专业设计、真实安装验证的空间智能知识**。

### 壁垒四：Builder-Studio-Life App 闭环生态

```
Builder（创作）→ Studio Core（运行：M300 / Cloud Studio）→ Life App（使用）→ Builder（迭代）
     ↑______________________________________________________________↓
                     XML(Builder) 流通 + 统一 Aqara 账号关联
                              ↑
                    Studio Cloud（纳管·下发·中继·推送，轻量层）
```

这个闭环中，每个环节都有数据产生：
- Builder 创作产生**方案设计数据**
- Studio Core 运行产生**设备状态 + 自动化触发数据**（优先本地；Cloud Studio 为试用/托管形态，与 M300 同 Runtime）
- Life App 使用产生**用户交互 + 习惯数据**（登录账号后与云端、AIOT 家庭数据可互通）
- Studio Cloud 产生**纳管与连接元数据**（不做设备状态主存）

**获客与转化补充（与第十二章一致）：** C 端可通过 **Builder 领取限时 Cloud Studio** → 同步 AIOT 设备 → **Life 登录后自动发现云 Studio** → 体验后 **转化 M300** 或续费云实例，使闭环同时承担增长飞轮。

**Workspace 与区域（补充）：** **Personal / 协作** 均在 Builder 显式存在；**Life 默认按 Personal + 权限护栏聚合**（§12.1.1），**不展示 Workspace 名词**；路由仍靠 `workspaceId` + Region。

三者（Builder / Studio Core / Life）合并，才能构成**完整的空间智能数据**。竞争者难以在三个环节同时拥有这样的数据。

### 壁垒五：社区网络效应（灵感广场飞轮）

```
Builder 越多 → 方案语料越丰富
方案越丰富 → AI 生成越好
AI 生成越好 → 更多 Builder 加入
Builder 越多 → 部署越多
部署越多 → 硬件销量增加
硬件销量增加 → Aqara 投入更多 AI / 设备研发
```

这是一个具备自我强化的飞轮。一旦跑起来，后入者需要同时在所有环节追赶。

## 第四章：面向世界模型时代——最深远的战略判断

### 4.1 什么是世界模型，为什么重要

**LLM（大语言模型）理解的是文字世界。**  
**世界模型（World Model）理解的是物理世界。**

- 李飞飞的 **Spatial Intelligence / Marble**：从图像理解空间结构、物体位置、三维关系
- Yann LeCun 的 **AMI / V-JEPA**：从观察预测未来物理世界状态，不依赖语言中转
- OpenAI Sora 的后续方向：视频生成 → 世界模拟 → 物理推理

这些模型未来能做到的事情，对智能家居行业意味着：

```
世界模型的能力                      对 Aqara 的价值
─────────────────────               ─────────────────────────────
从平面图理解三维空间布局             自动生成设备落位建议
从摄像头画面识别房间状态             实时理解"空间里发生了什么"
理解用户在空间中的行为模式           预测用户意图，优化自动化逻辑
对物理约束有感知（门不能开过去）     验证设备安装方案的物理合理性
从视频生成"如果灯光改变会怎样"      3D 方案预览，照明效果模拟
```

### 4.2 Aqara 的独特位置：真实物理世界的数据节点

**世界模型需要什么？**  
大量高质量的、**在真实物理环境中发生的、有标注的感知-行动数据**。

Aqara 的 Studio 运行在真实空间里。FP2 存在感应器感知真实人的运动轨迹。门窗传感器记录真实的家庭作息。温湿度传感器记录真实的环境变化。

> **Aqara 的设备网络，是全球规模最大的、有语义标注的真实家庭物理感知数据来源之一。**

这些数据 + XML(Builder) 方案语料（空间语义标注），将是训练空间智能世界模型的顶级数据集。

### 4.3 XML(Builder) 是世界模型的输入/输出格式

```
┌──────────────────────────────────────────────────────────────────────┐
│                    AI / 世界模型 交互层                               │
│                                                                      │
│  输入：                                                              │
│  • 户型图 / CAD / 照片（物理空间表示）                               │
│  • 用户需求（自然语言 / 行为数据）                                   │
│  • 空间类型语义（住宅 / 酒店 / 办公）                                │
│                                                                      │
│  ──────────────────────→  世界模型 / AI ←──────────────────────     │
│                                                                      │
│  输出：XML(Builder)                                                          │
│  • 空间图谱（Layout → Space Semantic Graph）                         │
│  • 设备落位方案（哪个位置，装什么设备，为什么）                       │
│  • 自动化逻辑（基于空间理解生成的行为规则）                          │
│  • 可视化图纸（供人类确认的设计图）                                  │
└──────────────────────────────────────────────────────────────────────┘
```

XML(Builder) 是**人类可读、机器可执行、世界模型可理解**的三合一格式。  
这个格式设计，使 Aqara Builder 在世界模型时代不需要重构——只需要接入更智能的模型。

### 4.4 三阶段演进路径

**第一阶段（当下 ~2027）：LLM 时代**
- 用 GPT / Claude 等大语言模型做意图理解和 XML(Builder) 生成
- 依赖预设模板 + 规则推理
- AI 是"写方案的助手"

**第二阶段（2027~2030）：多模态感知时代**
- 接入 Vision LM（GPT-4V 后代）
- 户型图照片 → AI 自动识别房间结构 → 直接输出 XML(Builder)
- 装修现场照片 → AI 识别已有设备位置 → 自动更新绑定关系
- AI 是"看图画方案的设计师"

**第三阶段（2030+）：世界模型时代**
- 接入 Marble / AMI 等空间智能世界模型
- 世界模型理解空间物理约束 + 用户行为模式
- 自动化规则从"工程师预设"变为"空间观察推断"
- XML(Builder) 从"人工填写的配置"变为"模型对空间的理解产物"
- AI 是"理解你家、替你思考的空间智能体"

### 4.5 与当下产品落地的一致性（避免「未来章」与「执行章」脱节）

无论处于哪一阶段，**XML(Builder) 的权威消费端**在工程上统一为：**Studio Core**（运行态 + 知识图谱）与 **Life 插件**（人机界面）；**Studio Web** 仅服务工程实施。  
**Cloud Studio** 与 **M300** 同为 Runtime 载体，便于试用、迁移与语料积累——与第十二章「本地化数字底座 + 云端轻增强」一致。

## 第五章：重构后的完整用户旅程（含空间可视化）

### 5.0 双轨旅程：与第十二章统一口径

全生态存在**两条并行主路径**，数据协议相同（XML(Builder) + 语义 ID + 插件），获客与转化不同：

| 轨道 | 典型用户 | 关键路径（摘要） |
|---|---|---|
| **B 端交付轨** | 方案商 → 安装商 → 业主/酒店客人 | Builder（**Workspace**）出方案 → 部署 **M300**（或企业托管 Studio）→ Studio Web **选 Workspace 上云绑定** → Life **扫码/装插件** |
| **C 端自驱轨** | 已有 Aqara 设备的个人用户 / DIY | Builder 注册 → **Personal Workspace + Region** → **限时 Cloud Studio（默认挂 Personal）** → **AIOT 同步插件** → **Life 登录**聚合 **Personal + LAN**（**不显式选 Workspace**）→ **M300 或云续费** |

> **5.1 大图默认描述 B 端交付轨**（签单与工程交付叙事最完整）。**C 端自驱轨的逐步动作与 UI 细节以第十二章 12.2～12.4 为准。**

### 6.1.1 Workspace：Personal 与协作型、数据驻留、计费与 Life 聚合（统一口径）

> **Workspace 是 Builder 控制台的组织与计费边界**；本地/云 **Studio 均挂在某一 Workspace 下**，再与该 Workspace 的 **Data Region** 所对应的 **Studio Cloud** 建立纳管与中继。

#### 两类 Workspace（当前产品设计）

| 类型 | 说明 | 邀请成员 | 删除 | 典型用途 |
|---|---|---|---|---|
| **Personal Workspace（个人工作区）** | **每个 Aqara 账号自动拥有且仅有一个**，系统预置 | **不允许邀请**（0 协作席） | **不允许删除** | 个人方案、试用、自家 **Cloud Studio 默认挂载点**、与 **Life「我家」心智**对齐 |
| **协作 Workspace（团队工作区）** | 用户**主动新建**（受套餐数量限制） | **可邀请**成员，**RBAC**，**可移除成员** | **允许删除**，但需满足 **其下已无任何 Studio 实例**（避免孤儿纳管） | 方案公司、项目组、安装商服务多个客户时的租户隔离 |

**多 Workspace 与身份：**

- 用户在 **Personal** 中**恒为 Owner**。
- 在**协作 Workspace** 中，可能是 **Owner**，也可能是**被邀请的成员**（仅具备被授予的权限）。

**免费 Cloud Studio（试用）默认归属：**

- **默认创建在用户的 Personal Workspace 下**，与「个人体验 → Life 默认可见」闭环一致；若产品允许「代客创建」等高级流程，可在协作 Workspace 中另建实例（B 端流程另述）。

**Workspace 核心属性（两类共用）：**

| 维度 | 说明 |
|---|---|
| **Data Region** | 创建时锁定 **CN / US / EU…**，驱动 **Studio Cloud** 区域路由与合规边界。 |
| **Studio 挂载** | Builder 中 **Studio / Cloud Studio 列表按 Workspace 分组**；Studio Web 绑定时 **必须指定其一**。 |
| **计费** | **AI Build 额度、协作席、项目上限等以 Workspace 为计费单元**；Personal 可有**基础免费配额**，协作 Workspace 消耗**团队套餐**。 |

**Studio Web：本地 Studio 上云绑定（不变）：**

```
Studio Web → Aqara 登录 → **Workspace 列表**（含 Personal + 有权限的协作 Workspace）
    → **必选其一**完成注册/纳管 → 对应该 Region 的 Studio Cloud
```

#### 设计取舍：Personal 为何「不可邀请」？要不要改成「可邀请但限人数」？

**当前方案（Personal 不可邀请）的优点：**

1. **隐私与责任边界清晰**：「个人工作区」= 仅本人，不会出现误把家庭云 Studio 共享给协作者的合规与客服风险。  
2. **与 Life 默认体验天然对齐**：C 端可把 Life 理解为 **账号 → Personal → 我家 Studio / 插件**，**不必理解组织**。  
3. **协作需求有明确出口**：必须协作时 **新建协作 Workspace**，计费与审计独立，利于 B 端开票与权限治理。

**备选方案（Personal 也允许邀请，仅卡人数）：**

- **优点**：少一个「建第二个 Workspace」的步骤，小团队可能在 Personal 里凑合协作。  
- **缺点**：**「个人」与「团队」语义混淆**，Personal 下的 Cloud Studio / 项目是否算「家庭」还是「公司」难以向用户与合规解释；**Life 侧仍要解决「哪些 Studio 上首屏」**，并不会因 Personal 可邀而消失。

**结论（文档建议）：** **保留 Personal = 不可邀请**；协作一律进 **协作 Workspace**。若未来数据证明小团队摩擦大，可评估 **「Personal 允许 1 名协作者」**等渐进放松，但需同步 **Life 可见性规则** 与 **数据分类**，不宜作为首发默认。

#### Life App 与 Personal 的关系（结论先行，§12.1.1 展开）

| 策略 | 说明 |
|---|---|
| **默认（业主 / 个人用户）** | **「生活」首屏聚合 Personal Workspace 下的 Studio** + **同账号 LAN 发现的本地 Studio** + **扫码/安装的插件**；**不默认展示**协作 Workspace 下的 Studio，**避免把客户现场与自家混在一起**。 |
| **协作者护栏（必须）** | 若某用户 **Personal 下无任何云实例**，但在 **协作 Workspace** 中被授予 **Life 可见 / 控权**，则 **仍须在 Life 中展示对应 Studio**（可 **不显式出现 Workspace 名词**，仅用「项目别名 / Studio 名称」分组），否则纯协作者 **无法用手机履职**。 |
| **进阶（安装商等）** | 「我」中可选 **「全部可访问的 Studio」** 或 **工作区筛选**，**非 C 端主路径**。 |

| 端 | 是否展示 Workspace 名词 |
|---|---|
| **Builder / Studio Web** | **是**，且须区分 Personal / 协作 |
| **Life（默认）** | **否**；后台按 `workspaceId` + 聚合规则路由 |
| **Life（可选）** | 「我」中工作区/组织管理 **渐进披露** |

### 6.3 分行业价值主张

#### 住宅精装交付（开发商 / 设计院）

- **额外价值**：能耗分析报告（房间使用率 × 能耗数据）→ 物业提效

#### 医养 / 适老化（未来重要方向）
- **痛点**：老年用户对复杂 App 完全无感，但对安全感和便利性需求极强
- **Builder 价值**：一键生成极简 UI（大字体 + 场景按钮）+ 安全告警逻辑
- **额外价值**：照护者远程查看空间地图状态 → 老人安全可视化

## 第七章：核心判断更新

在 v1 的六个判断基础上，新增以下关键判断：

### 判断七：空间可视化是必做，不是"锦上添花"

用户的反馈已经非常明确：让方案"看得见"，是签单的决定性因素。  
设备落位平面图、联动路径动画、App UI 原型——这三个可视化产物，应该成为 AI Build 的**默认输出**，而不是可选项。

实现路径（分阶段）：
- **Phase 1**：基于 XML(Builder) 空间语义，在 Canvas 上自动生成 SVG 设备落位示意图（不需要精确坐标，只需要房间 + 设备类型）
- **Phase 2**：支持导入户型图（PNG/PDF），AI 识别房间轮廓，将设备图标叠加在真实平面图上
- **Phase 3**：接入 Vision LM，从照片自动识别房间，生成设备落位建议

### 判断八：Studio Web 要做到"施工图级别"的工程交付

当前 Studio Web 是"配置 UI"。  
未来 Studio Web 应该是**智能空间的工程管理平台**：
- 设备安装进度追踪（哪台设备已安装 / 待安装 / 调试中）
- 绑定完成率仪表盘（方案中 30 台设备，已绑定 28 台，待处理 2 台）
- 交付验收报告（自动生成 + 业主签字功能）
- 售后服务入口（设备故障 → 工单 → 派工）

### 判断九：Life App 要有空间地图视图

住户不只想控制设备，他们想**看到自己的家**。  
一张户型平面图，上面有实时状态标注：
- 客厅灯 → 亮（橙色光晕）
- 主卧窗帘 → 关闭（蓝色）
- 玄关门 → 关闭（绿色）
- 厨房烟雾传感器 → 正常（绿色）

这个界面的价值：
- 住户一眼看到家里的整体状态（不用逐个设备查）
- 异常状态有位置感（"厨房"的烟雾，不是一条抽象告警）
- 深度绑定了住户和空间的情感连接（"这是我家的智能化方案"）

### 判断十：从第一天开始，为世界模型输入准备数据结构

未来接入 Marble / AMI 等世界模型，需要的数据：
- **空间几何**：房间形状、门窗位置、家具布局（建议 XML(Builder) 增加 `geometry` 字段）
- **设备位置**：设备在空间中的 XYZ 坐标或相对位置描述
- **行为轨迹**：用户在空间中的移动模式（FP2 数据的匿名聚合）
- **事件日志**：何时触发了哪条自动化，效果如何

这些数据**现在就要以结构化格式存储**，即使今天暂时没有世界模型来消费。

### 判断十一：统一 Aqara 账号 + Life「生活·市场·商城·我」是商业闭环的必要条件

**账号层：** Life App **默认引导登录**与 **Aqara Home（AIOT）同一账号体系**。登录后用户才能：发现账号下的 **Cloud Studio**、走 **Studio Cloud** 中继与推送、在 **商城** 完成设备/M300 转化、与历史家庭数据互通。未登录仅适合 **纯 LAN 本地**场景，不做为主路径。

**壳层与转化：** Life 不是「无限堆功能的大 App」，而是 **插件引擎 + 固定四 Tab**。**商城** 与 **市场** 分工明确：市场卖「方案/插件内容」，商城卖「硬件与升级路径（含 M300）」——两者共同承担 **GMV 与 Cloud→Local 转化**，与第十二章漏斗一致。

**与 B 端场景的关系：** 酒店扫码、访客 Token 等 **可无 Aqara 账号**；但 **业主与增长主路径**仍以账号为枢纽，避免「装了插件却永远不进商业闭环」。

### 判断十二：Personal / 协作 Workspace 分工，Life 默认聚合 Personal + 协作者护栏

- **Builder**：每账号必有且仅有一个 **Personal Workspace**（**不可删、不可邀请**）；协作需求进 **协作 Workspace**（可邀请、RBAC、**无 Studio 可删**）。**免费 Cloud Studio 试用默认挂在 Personal**。**Data Region** 按 Workspace 锁定，驱动 **Studio Cloud** 路由。
- **Studio Web**：上云绑定须 **选择 Workspace**（Personal 或协作）。
- **Life App**：**界面不默认出现 Workspace**；**默认「生活」聚合 Personal 下 Studio + LAN + 插件**；**协作 Workspace 下已授权 Studio 在 Personal 为空时必须仍可出现**（§12.1.1 护栏）。安装商等可在「我」中 **可选**查看全部可访问 Studio。
- **备选「Personal 也可邀请仅限人数」**：可作为远期放松，但会模糊个人/团队边界与合规叙事；**首发不推荐**，见 §6.1.1 取舍说明。

### 10.0 平台核心架构原则：三元素解耦

**这是整个平台最重要的设计哲学，所有产品决策都从这里出发：**

| 元素 | 职责 | 存储位置 | 关系 |
|---|---|---|---|
| **XML(Builder)** | 方案的灵魂：空间图谱 + 设备定义 + 自动化逻辑 | Aqara Cloud + 本地 | 可部署到**任意** Studio |
| **Studio** | 方案的身体：本地运行时 + 真实设备网关 | M300 本地 / Cloud Studio | 可运行**任意** XML(Builder) |
| **Plugin** | 方案的眼睛：特定角色的 App UI 视角 | Aqara Cloud | 运行时**动态发现** Studio |

**三者之间是"多对多"的松耦合关系：**

```
一个 XML(Builder) → 部署到多台 Studio
    示例：连锁酒店同一标准方案，复制部署到 100 间客房（每间独立 M300）

一个 XML(Builder) → 构建多个插件变体
    示例：业主插件 + 家人插件 + 访客插件 + 运维插件（同一 XML(Builder)，4 种视角）

一个插件 → 运行时动态连接 Studio（不在构建时绑定！）
    示例：扫码后 Life App 自动搜索运行该 XML(Builder) 的 Studio，再建立连接

一个插件 → 可同时适配多台 Studio（同 XML(Builder) 多部署场景）
    示例：连锁咖啡品牌的"店员插件"，在任意门店部署后均可使用
```

**这个解耦架构解锁了三个关键能力：**

1. **XML(Builder) 是可移植方案包**：在 Builder 上设计，在任意 Studio 上运行，不绑定特定硬件
2. **插件是通用视角**：同一插件 QR 可跨不同场地复用（连锁品牌、多门店、多房间）
3. **预览模式原生支持**：即使 XML(Builder) 未部署到任何 Studio，插件也能完整展示方案 UI

**Studio 的本质：空间知识图谱（Space Knowledge Graph）**

> Studio 不仅是"设备网关"，它是一个**以图谱方式组织的空间知识库**。  
> 每一个对象（空间、设备、自动化、场景）都有全局唯一的语义 ID，节点之间有关系边连接。  
> 物理设备地址（Zigbee MAC、Matter UUID 等）是实现细节，**对外只暴露语义 ID**。

```
Studio 知识图谱（示意）

节点（Objects，每个有唯一语义 ID）
├── 空间节点：living_room, bedroom_master, kitchen
├── 设备节点：main_light_lr（客厅主灯）
│       ├── 物理绑定：zigbee://AA:BB:CC:DD（部署时建立）
│       └── 功能点：on_off, brightness, color_temp
├── 自动化节点：auto_away_mode（离家模式）
└── 场景节点：scene_movie（观影场景）

边（Relations）
├── main_light_lr  → belongsTo  → living_room
├── auto_away_mode → controls   → main_light_lr, ceiling_fan_lr
└── scene_movie    → activates  → [main_light_lr@brightness=20%, projector@on]
```

**关键：XML(Builder) 中的 ID 是"语义 ID"，不是物理设备地址**

```
Builder UI 设计时（绑定语义 ID）：
    [客厅灯开关卡片] → 绑定 ID: "main_light_lr"（纯语义，无物理依赖）

导入到 Studio A（家里 M300）：
    Studio A 映射："main_light_lr" → zigbee://AA:BB:CC:DD

导入到 Studio B（样板间 M300）：
    Studio B 映射："main_light_lr" → matter://device-xyz（不同设备，同一语义）

Plugin 查询："main_light_lr 的当前状态是什么？"
Studio 解析：语义 ID → 物理设备 → 实际状态 → 返回给 Plugin UI
```

> **这是预览模式（Preview Mode）可行的深层原因：**  
> Plugin 渲染的是语义结构（ID + 类型 + 位置关系），无需物理设备就能展示完整 UI 骨架。  
> 连接 Studio 后，语义 ID 被解析为真实设备状态，UI 从"静态骨架"变为"动态实控"。

### 10.1 插件的本质：一张"空间通行证"

**关键认知升级：**

> 旧认知：Plugin = 连接特定 Studio 的授权连接  
> 新认知：Plugin = `XML(Builder) 方案 × 角色配置 × UI 主题`（不包含任何 studioId！）  
>
> **哪台 Studio 正在运行该 XML(Builder)，Plugin 就能在运行时连接到哪台。**

**插件的内部构成（存于 Aqara Cloud，以 pluginId 索引）：**

```
Plugin 配置
├── XML(Builder)Id          方案身份（我是为哪个 XML(Builder) 方案服务的）
├── roleConfig      角色权限（可见页面 / 可控设备 / 有效期 / 时间窗口）
├── themeId         UI 主题（视觉风格）
├── scopeConfig     空间范围（全部 / 指定楼层 / 指定房间）
└── ← 无 studioId  关键！不在构建时绑定 Studio，运行时动态发现
```

**QR Code / 分享链接的格式（极简）：**

```json
{
  "type": "aqara-plugin",
  "pluginId": "plugin-abc123",
  "accessToken": "tok_xxx"   // 可选：私有插件的身份凭证（证明有权使用此角色，不是 Studio 密钥）
}
```

> `accessToken` 的作用：**证明"我有权以此角色使用这个方案"**，而非"我要连接到这台 Studio"。  
> 哪台 Studio 在运行该 XML(Builder)，由 Life App 运行时自动发现——不是写在 QR 里的。

**同一 XML(Builder) 的多角色插件构建模型：**

```
XML(Builder) 方案（数据源，XML(Builder)Id = "XML(Builder)-cozy-3br"）
    │
    ├── 角色 A：Owner（业主）
    │       权限：全部 · 主题：家居温暖 · 有效期：永久
    │       → pluginId: plugin-001-owner  accessToken: owner-tok（永久）
    │
    ├── 角色 B：Family（家庭成员）
    │       权限：公共区域 + 本人房间 · 主题：家居温暖 · 有效期：永久
    │       → pluginId: plugin-001-family  accessToken: family-tok（永久）
    │
    ├── 角色 C：Guest（访客）
    │       权限：客厅场景 · 主题：极简白 · 有效期：48 小时
    │       → pluginId: plugin-001-guest  accessToken: guest-tok（48h，可撤销）
    │
    └── 角色 D：Cleaning（清洁人员）
            权限：清洁模式触发 · 主题：功能型 · 时间窗口：工作日 9-18 点
            → pluginId: plugin-001-cleaning  accessToken: clean-tok（时段有效）
```

**UI 主题选项（与权限无关，独立叠加）：**

| 主题名 | 适用场景 | 视觉特征 |
|---|---|---|
| 家居温暖 | 住宅业主 | 暖色调，卡片式，图标动效 |
| 极简白 | 访客/临时 | 纯白，大按钮，只显示场景 |
| 专业深色 | 办公/商业管理 | 深色，数据密集，状态仪表盘 |
| 酒店礼宾 | 酒店客人 | 品牌色，宾馆风格，一键场景 |
| 适老版 | 老人用户 | 超大字体，高对比，极简操作 |
| 工程调试 | 安装商 | 显示设备 ID/协议/状态原始数据 |

### 10.4 权限架构：三层防御，缺一不可

仅仅"UI 上不显示"是不够的——用户可能用工具直接发 HTTP 命令给 Studio。  
权限必须在三层都完整闭合。

```
┌────────────────────────────────────────────────────────────┐
│  第一层：连接权限（Connection Auth）                        │
│  问：这个 Life App 实例，是否有权连接到这个 Studio？         │
│  手段：Role Token（扫 QR Code 时获得，有时效和撤销）        │
│  位置：Studio 的会话层验证                                  │
├────────────────────────────────────────────────────────────┤
│  第二层：UI 范围权限（View Scope）                          │
│  问：这个角色，可以看到哪些页面/设备/场景？                  │
│  手段：Plugin Variant 配置文件（构建时固化在插件里）         │
│  位置：Life App 插件渲染层（客户端）                        │
├────────────────────────────────────────────────────────────┤
│  第三层：指令执行权限（Command ACL）                        │
│  问：这个 Role Token，是否允许执行这条命令？                 │
│  手段：Studio 端命令级 ACL（每条命令都校验）                │
│  位置：Studio 指令处理层（服务端，不可绕过）                │
└────────────────────────────────────────────────────────────┘
```

**第三层是安全底线：**  
即使有人修改了客户端代码、伪造了 UI、或者直接 HTTP 请求 Studio，Studio 会在收到每条命令时验证 Role Token 的命令权限，不符合的直接拒绝。

**Role Token 的生命周期：**

```
Builder 生成插件变体
    │
    └── Aqara Cloud 创建 Plugin 配置 + accessToken
        （含：角色 ID，权限列表，有效期，XML(Builder)Id，无 studioId）
            │
            └── QR Code 编码：{ pluginId, accessToken }（极简，不含 Studio 信息）
                    │
                    └── 用户扫码 → Life App 保存 { pluginId, accessToken }
                            │
                            ├── Life App 获取 Plugin 配置：XML(Builder)Id, roleConfig, themeId
                            │
                            ├── Life App 发现 Studio（10.3 协议）→ 连接 → 签发 Session Token
                            │
                            ├── 有效期到期 → accessToken 失效 → Life App 显示"访问已过期"
                            │
                            └── 业主手动撤销 → accessToken 立即失效（踢出登录）
```

#### 场景一：在家（同一局域网）

```
用户回家 → 连上家里 WiFi
    → Life App 检测到 Studio M300 在同一网络
    → LAN 直连：ws://192.168.1.100:7890
    → 指令在本地闭环，无需经过云端
    → 即使家里宽带断了，依然可以控制所有设备
    → 响应时间 < 50ms，体感"即时响应"
```

#### 场景二：外出（离开家庭网络）

```
用户出门 → 切换至 4G/5G 或其他 WiFi
    → Life App 检测 LAN 发现失败
    → 自动切换至云中继路径
    → Aqara Cloud 维护 Studio M300 的持久隧道连接
    → 指令路径：Life App → Aqara Cloud → Studio M300
    → 正常网络延迟 200-500ms，体感略有延迟
    → 所有权限、规则完全一致，只是路径不同
```

#### 场景三：在家但宽带断了

```
家里停电（仅宽带断，本地局域网还在）
    → Life App 在同一 WiFi → LAN 直连成功
    → 本地 Studio 完全可控
    → 云端功能（远程推送、语音助手、第三方联动）暂不可用
    → 用户日常控制不受影响
    
家里完全断电（M300 和路由器都断）
    → Life App 无法连接任何路径
    → 显示设备最后已知状态（缓存）
    → 提示"Studio 离线，显示上次状态"
    → 断电恢复后自动重连，状态同步
```

#### 场景四：酒店客人

```
客人入住 → 前台/房卡二维码包含 { pluginId: hotel-guest, accessToken: room-301-xxx }
    → 客人扫码 → Life App 加载插件配置（XML(Builder)Id: hotel-standard-room, 客人角色）
    → 插件渲染 Preview Mode（即刻可见，UI 完整呈现）
    → Life App 后台发现 Studio-301（酒店 WiFi 内 mDNS 或云端注册表）
    → 连接 Studio-301 → 切换为实控模式
    → 客人在酒店 WiFi 内 → LAN 直连（快速响应）
    → 客人外出回来，到大堂开空调 → 云中继（仍可远程预热）
    → 退房后 accessToken 自动失效 → Life App 显示"此房间访问已结束"
    → 客人无需注册 Aqara 账号（accessToken 独立于账号体系）
```

#### 场景五：多用户同时在家

```
业主 + 配偶 + 孩子 同时在家
    → 三个 Life App 实例同时连接同一 Studio
    → Studio 维护多个 WebSocket 连接
    → 业主开了客厅灯 → Studio 广播状态变更给所有在线的 Life App
    → 所有人的 App 实时刷新（不需要手动刷新）
    → 孩子尝试关掉客厅空调 → Studio 验证孩子 Session Token 权限
       → "孩子"角色无权控制公共区域空调 → 指令被拒绝
       → Life App 显示"没有权限执行此操作"
```

### 10.9 设计总结：四个核心原则

**原则一：插件与 Studio 解耦，运行时动态发现**

> Plugin 不绑定特定 Studio 实例。Plugin = XML(Builder) × 角色 × 主题。  
> Studio 是在 Life App 加载插件后，通过 LAN mDNS 和云端注册表自动发现的。  
> 这使得同一插件模板能在链式复制场景（如连锁酒店）中零成本复用。

**原则二：预览模式是一等公民，不是"未完成态"**

> 即使 XML(Builder) 未部署到任何 Studio，插件也能完整渲染方案 UI。  
> 预览模式是销售工具、演示工具、确认工具——不是等待状态。  
> 这让"先看效果、再安装硬件"成为完整的产品体验路径。

**原则三：角色即视角，而非账号**

> 传统智能家居用"账号系统"管权限。Aqara Builder 用"角色 + accessToken"管权限。  
> 区别：accessToken 可无账号使用（酒店客人不需要注册 Aqara），可精确时效，可立即撤销。  
> 这让 B 端场景（酒店/民宿/办公）的用户管理成本极低。

**原则四：LAN 优先，云中继是降级路径，离线是最后防线**

> 智能家居的控制不应依赖云端可用性。  
> LAN 直连保证"家里的网即使断了也能用"，这是对用户的基本承诺。  
> 云中继是方便，不是基础能力。权限在 Studio 服务端闭合，UI 只是展示过滤。

### 10.10 Life App 作为空间操作系统：插件引擎 + 固定壳层（与第十二章对齐）

**设计目标：** Life 是 **插件引擎**，但**不是**「无壳纯白板」——应有**稳定的 App 壳**承载增长与转化：**底部 [生活][市场][商城][我]** 始终存在；**「生活」**内才是插件主内容区（单插件直达 / 多插件卡片列表，见 §12.4）。

**与账号的关系（主路径）：** **登录 Aqara 账号**后，Life 默认聚合 **Personal Workspace** 下云实例与已注册 M300，并走 **Studio Cloud** 中继、推送与插件下发；**协作 Workspace 授权**须满足 **§12.1.1 协作者护栏**。**未登录**仅 **LAN 发现本地 Studio**。**Workspace 名词**默认不出现在 UI（详见 **§12.1.1**）。这与「酒店客人扫码、访客 Token」等**无账号场景**并存：后者走 **pluginId + accessToken**，不经过「我的 Studio 列表」。

**首次打开体验（摘要，与 §12.3 一致）：**

```
下载 Life App → 欢迎页
    ├── [登录 / 注册 Aqara 账号]     ← 主路径（与 Aqara Home / AIOT 互通）
    └── [本地模式（不登录）]         ← 次路径（高级/隐私场景）

登录后：
    ├── 若账号下已有 Cloud Studio（如 Builder 领取试用并已部署）
    │       → 「生活」可直接展示该空间插件 / 引导安装插件
    ├── 若仅有 M300 已注册
    │       → 优先 LAN，失败则云中继
    └── 若尚无任一 Studio
            → 引导：去 Builder 领云 Studio / 扫码装插件 / 打开「市场」

未登录：
    → 「生活」内 mDNS 扫描；无 Studio 时提示登录以使用云端能力
```

**插件安装后的状态流转（技术层不变，发现路径随登录态变化）：**

```
安装插件（扫码或从「市场」）
    │
    ├── Preview Mode：语义 UI 完整，无 Studio 或无权连接时
    ├── 已登录：优先匹配账号下 Cloud Studio / 注册表中的目标实例
    ├── 未登录：仅 LAN 匹配；不展示 Cloud Studio
    └── 连接成功 → 实控模式（Studio 知识图谱解析语义 ID，§10.0）
```

**「市场」与「商城」分工（避免能力堆叠）：** 「市场」= 方案与插件内容分发；「商城」= **设备、M300、配件、服务包** 等 **GMV**。服务类插件（天气、能耗报告等）仍在插件体系内，**不替代商城作为硬件主入口**。

**Life App 插件的三种类型：**

| 类型 | 描述 | 是否需要 Studio | 示例 |
|---|---|---|---|
| **空间插件（Spatial Plugin）** | 代表一个物理空间，需连接 Studio 实控 | 是（Preview Mode 不需要） | 我的家 · 酒店房间 · 公司楼层 |
| **服务插件（Service Plugin）** | 独立功能服务，无需 Studio | 否 | 天气 · AI 助手 · 能耗报告 · 内容资讯（**硬件购买以「商城」Tab 为主入口**） |
| **组态插件（Dashboard Plugin）** | 用户自由设计的控制大屏，绑定 Studio 语义 ID | 是 | 自定义仪表盘 · 全屋状态总览 |

**组态插件（Dashboard Plugin）的设计逻辑：**

> 类比本地 Studio 的"组态大屏"功能——用户在 Life App 上也可以自由设计控制界面。  
> 用户添加卡片（灯光控制、温度显示、摄像头画面、快捷场景按钮……）  
> 每张卡片绑定到 Studio 知识图谱中的语义 ID（设备功能点 / 自动化 / 场景）  
> 这让高级用户（极客/Prosumer）可以在 Life App 上实现完全个性化的控制界面

```
Dashboard 插件构建流程（Builder 或 Life App 内）：

[添加卡片 +]
    ├── 灯光开关卡片 → 绑定 Studio ID: main_light_lr
    ├── 温度显示卡片 → 绑定 Studio ID: temp_sensor_bedroom
    ├── 场景按钮卡片 → 绑定 Studio ID: scene_movie
    └── 摄像头卡片   → 绑定 Studio ID: camera_entrance

布局自由拖拽 → 保存 → 生成 Dashboard 插件
→ 可在 Life App 安装使用 / 分享给他人
```

**用户自定义 Life App 的三条路径：**

```
路径 1：插件市场（最轻松，适合普通用户）
    浏览公开模板 → Preview Mode 体验 → 喜欢就安装
    → 安装硬件后连接 Studio → 实控激活

路径 2：扫码安装私有插件（最常见 B2B2C）
    安装商交付时给 QR → 扫码 → 立即连接 Studio
    （酒店入住 / 长租入住 / 办公室新员工 → 同一路径）

路径 3：Builder 自建（Pro 用户 / 极客）
    在 Builder 上设计 XML(Builder) 方案 → 生成插件 → 扫码安装
    → 自己创作，自己使用，可选发布到插件市场
    → 构建 Dashboard 插件，实现极度个性化控制界面
```

| 对比系统 | Life App | 插件 | Studio |
|---|---|---|---|
| 手机操作系统 | iOS / Android | App | 云服务 / 本地数据库 |
| 微信 | 微信 | 小程序 | 小程序服务端 |
| Home Assistant | HA 前端 | Integration / Dashboard | HA 后端 + 设备 |
| 浏览器 | Chrome | 网页 / PWA | Web 服务器 |

> 与手机 App 不同的是：空间插件连接的是你的**物理现实**，每次控制都产生真实的物理效果。

## 第十一章：开放平台商业模型——XML(Builder) 即产品，Studio 即基础设施

> 三元素解耦不仅是技术架构，更重新定义了商业模型的边界。  
> **XML(Builder) 是可售卖、可订阅、可授权的空间智能软件资产。**  
> **Studio Core（M300 / Cloud Studio）是承载 XML(Builder) 运行的数字底座；Studio Cloud 是轻量纳管与中继层。**  
> **Plugin 是连接用户与空间的交互入口；Life App 的 [市场][商城] 分别承载「内容变现」与「硬件 GMV」。**  
> **Builder 的 Workspace 是 SaaS 计费与协作租户单元**（含 **Data Region**），套餐以 Workspace 为主（AI Build、协作席位等）。  
>
> **执行优先级与阶段裁剪仍以第十二章为准**（例如当前阶段不做白标 Life、不做过重云端业务）。

**平台服务的四类用户（核心用户画像，与第五章双轨一致）：**

| 用户类型 | 描述 | 使用路径 | 商业价值 |
|---|---|---|---|
| **B 端方案商/安装商** | 用 AI 设计方案，为 C 端客户交付 | Builder（**Workspace**）→ XML(Builder) → **M300** → Studio Web **选 Workspace 上云绑定** → 客户 Life 装插件 | **Workspace 套餐** + 方案授权 + **硬件批量** |

### 11.0 插件的三种类型：Spatial · Service · Dashboard

在规划商业模型之前，先明确插件的类型——不同类型的商业逻辑完全不同。

**类型 1：空间插件（Spatial Plugin）**

```
定义：代表一个物理空间，需要连接 Studio 实控
数据基础：来自 XML(Builder)（空间图谱 + 设备 + 自动化）
状态模式：Preview（无 Studio）→ Connected（有 Studio）
典型场景：家庭智能控制 · 酒店客房 · 办公室楼层 · 样板间展示
商业来源：方案包销售 / 连锁部署授权 / 安装后激活
```

**类型 2：服务插件（Service Plugin）**

```
定义：独立功能服务，无需 Studio，任何 Life App 用户都可使用
数据基础：来自 Aqara 云服务或第三方 API
典型示例：天气与室外环境 · AI 助手 · Aqara 设备商城 · 能耗报告 · 家居建议
商业来源：第三方开发者分成 / Aqara 官方产品引流 / 订阅增值服务
```

**类型 3：组态插件（Dashboard Plugin）**

```
定义：用户自由设计的控制大屏，绑定 Studio 知识图谱中的语义 ID
设计工具：Builder 上设计（类似 Studio 本地的组态大屏功能，搬到 App 上）
卡片类型：设备控制卡片 · 状态显示卡片 · 场景按钮 · 摄像头画面 · 数值图表
目标用户：Pro 用户 / 极客 / Prosumer（终极个性化控制需求）
商业来源：Pro 版功能订阅 / Dashboard 模板市场付费分成
```

> **三种类型的关系：**  
> 普通用户从"空间插件"入门，享受开箱即用的体验。  
> 活跃用户加装"服务插件"，丰富日常信息获取。  
> 极客用户创建"组态插件"，实现完全个性化控制界面。  
> 三种类型共同构成 Life App 的完整生态。

```
┌─────────────────────────────────────────────────────────────────┐
│  Studio Core（本地，持续运行）                                    │
│                                                                 │
│  • 插件化框架：所有功能以插件形式存在，可动态加载/卸载/实例化     │
│  • 知识图谱引擎：维护空间语义图（对象 + 关系 + 语义 ID）          │
│  • 物理设备网关：Zigbee / Matter / BACnet / Modbus 接入          │
│  • 规则引擎：自动化逻辑执行（触发器 / 条件 / 动作）               │
│  • 本地 API：WebSocket（实时状态）+ HTTP（查询/控制）             │
│  • mDNS 广播：让 Life App 自动发现本机 Studio                    │
│                                                                 │
│  部署形态：                                                      │
│  ├── M300（本地硬件）：持久运行，断网可用，最优性能               │
│  └── Cloud Studio（云端虚拟实例）：试用/托管场景，按需收费         │
│                                                                 │
│  ── GUI 层（Studio Web）────────────────────────────────────────│
│  • Studio Core 的工程实施 GUI，仅此而已                          │
│  • 面向：安装商、IT 管理员、工程调试、高级用户自配               │
│  • 功能：设备绑定 · 自动化配置 · 组态大屏设计 · XML(Builder) 导入/导出   │
│                                                                 │
├─────────────────────────────────────────────────────────────────┤
│  Studio Cloud（云端，轻量管理层）                                 │
│                                                                 │
│  三件事原则（不做第四件）：                                       │
│  ① 管理：纳管已连接的 Studio（Cloud Studio 实例 + 注册的 M300）   │
│  ② 分发：向本地 Studio 下发插件、配置更新、XML(Builder) 方案              │
│  ③ 中继：为本地 Studio 提供远程访问隧道 + 事件推送通道            │
│                                                                 │
│  不做：不存储设备实时状态，不执行自动化，不做业务决策             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

> **Cloud Studio 与 M300 使用完全相同的 Studio Core Runtime。**  
> 用户从 Cloud Studio 升级到 M300 时，XML(Builder) 方案、知识图谱、插件配置一键迁移，零重新配置。

### 12.1.1 Workspace：Personal / 协作、上云绑定与 Life 聚合规则（SSOT）

**与 §6.1.1 一致，此处只固定「执行层必须一致」的结论。**

**1）两类 Workspace**

- **Personal Workspace**：**每账号唯一**、**不可删除**、**不可邀请成员**；用户恒为 Owner；**免费 Cloud Studio 试用默认挂在此**。  
- **协作 Workspace**：用户**新建**；**可邀请**、**RBAC**、**可移除成员**；**无 Studio 时可删除**；用户可能是 Owner 或成员。

**2）Studio Web 上云绑定**

- 登录后列出 **Personal + 有权限的协作 Workspace** → **必选其一**绑定本地 Studio → 对应该 Workspace **Data Region** 的 Studio Cloud。

**3）Life App：不呈现 Workspace 名词时的「聚合规则」**

**默认主路径（个人用户）：**

```
Aqara 账号登录 Life
    → 后端聚合（默认「生活」首屏）：
        ├── Personal Workspace 下的 Cloud Studio / 已注册 M300
        ├── 同账号、同手机的 LAN 直连 Studio（与 Workspace 展示无关）
        └── 已安装的插件（含扫码，可能指向任意 Studio）
    → **不默认列出**「仅挂在协作 Workspace 下」的 Studio
        （避免业主首页混入客户项目、酒店总部实例等）
```

**协作者护栏（必须实现）：**

- 若用户 **Personal 下没有可展示的 Studio**，但在某一 **协作 Workspace** 中被授予 **Life 侧可见/可控**权限，则 API **必须返回这些 Studio**，否则协作者 **Life 空白、无法履职**。  
- UI **仍可不写 Workspace**：用 **Studio 别名 / 项目名称** 列表即可；可选在「我」里标注「工作项目」来源。

**进阶用户（安装商 / 多客户）：**

- 「我」→ **「全部 Studio」或按来源筛选**（含协作 Workspace），**非首屏默认**。

**4）与「酒店扫码 / Token」**

- 无 Aqara 账号的访客路径 **不经 Workspace**；与上表并行存在。

**5）原则重申**

- **Builder / Studio Web = 组织面（含 Personal/协作类型）**；**Life = 体验面（默认 = Personal + LAN + 插件，协作 Studio 靠权限护栏与进阶入口）**。

### 12.5 Studio 事件推送：轻量化设计方案

**问题根源：** 如果 Studio 把所有事件广播给所有连接的 Life App，流量和计算会很重。

**解决方案：角色订阅过滤（Role-Filtered Pub/Sub）**

```
Life App 连接 Studio 时，注册订阅范围：
    Session 建立 → Life App 发送订阅声明：
    {
      sessionId: "sess_xxx",
      roleId: "guest",
      subscribeIds: ["main_light_lr", "ac_unit_lr", "scene_movie"]
      // 仅订阅角色权限范围内的对象，其他对象的状态变化不推送
    }

Studio 事件触发时（如：main_light_lr 状态变化）：
    → 查询订阅注册表："谁订阅了 main_light_lr？"
    → 找到 sess_001（业主）、sess_002（家人）、sess_004（访客）
    → sess_003（清洁人员）未订阅 → 不推送
    → 只向 sess_001/002/004 发送状态更新事件
```

**Studio 的连接分层（避免单 Studio 过重）：**

```
连接类型分层：
├── 控制连接（Control Session）：Life App 的全双工 WebSocket
│   → 实时状态订阅 + 控制指令发送
│   → 每个角色一个连接，按订阅过滤推送
│
├── 只读连接（Read-Only Session）：Studio Web 的状态监控
│   → 较低频率轮询（5秒一次），不占用实时推送资源
│   → 工程调试模式可升级为全量订阅
│
└── 隧道连接（Tunnel Session）：Studio Cloud 的持久中继隧道
    → 一条隧道服务所有远程客户端
    → Studio Cloud 做事件分发（不做业务逻辑）
```

**轻量化的三个原则：**
1. **按需订阅**：Life App 只告诉 Studio 它关心什么，Studio 只推它关心的
2. **Studio Cloud 不做状态**：远程访问通过隧道透传，不在云端缓存设备状态
3. **本地优先**：LAN 直连时完全不经过 Studio Cloud，零云端负担

### 12.6 旧系统迁移路径：AIOT Platform → Studio

> 已有 Aqara Home App + 微服务 AIOT 平台的用户，需要平滑迁移到本地 Studio 战略。  
> 不是强制切换，而是**渐进迁移，允许并行运行**。

**迁移插件（Studio Web 中的一个内置插件）：**

```

迁移后：
    ├── 本地 Studio 成为主控（低延迟、断网可用）
    ├── 原 AIOT 平台保留只读访问（降级为数据备份）
    └── 原 Aqara Home App 可继续使用（过渡期）
```

**并行运行策略（平稳过渡，不强制切换）：**

```
阶段 1（并行）：
    本地 Studio 控制 ← 主路径（大部分设备）
    Aqara Home App  ← 仍可使用（云端路径）

阶段 2（主迁移完成）：
    本地 Studio 控制 ← 全部设备
    Life App         ← 替代 Aqara Home App

阶段 3（旧平台下线）：
    本地 Studio      ← 唯一运行时
    旧平台数据       ← 归档备份
```

### 12.8 当前阶段商业聚焦（不做的事）

> **战略收敛的另一面：明确告诉自己现在不做什么。**

**现阶段不做：**
- ❌ 白标 Life App（等 Phase 2 验证完核心路径）
- ❌ Life App SDK（等有企业客户主动需要时）
- ❌ XML(Builder) Open Schema 对外发布（先内部稳定，再开放）
- ❌ 全面替换 Aqara Home App（渐进迁移，用户主动选择）
- ❌ 复杂的多 Studio 管理（先单 Studio 跑通，再扩展）
- ❌ AI Plugin Generator（先把插件框架做稳，再 AI 化）

**现阶段必须做：**
- ✅ XML(Builder) 语义 ID 体系（这是所有后续功能的基础，必须先稳定）
- ✅ Studio Core 插件架构（一旦确定就不要大改，后续都在上面叠）
- ✅ **Personal Workspace + 协作 Workspace 模型 + Data Region；Studio Web 选 Workspace 绑定 Studio Cloud**
- ✅ **Life 默认 Personal 聚合 + 协作者护栏（见 §12.1.1）**
- ✅ Life App 插件引擎（扫码 → 安装 → Preview → 实控，这个最小流程）
- ✅ Studio-Life App 连接协议（LAN + Cloud Relay，协议定下来）
- ✅ Builder → XML(Builder) → Studio 的完整链路（哪怕 UI 粗糙，流程必须通）
